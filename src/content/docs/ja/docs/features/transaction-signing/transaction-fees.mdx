ライブラリが手数料決定を自動処理する場合がありますが、Xahauネットワークで独自の統合を構築する場合、トランザクション、送信元、宛先アカウントに基づいた動的な手数料決定を実装する必要があるかもしれません。

トランザクションの送信者は、特定のトランザクションタイプに対して呼び出されるHooksに必要な手数料を支払わなければなりません。Hooksは送信元アカウントと宛先アカウントの両方に存在する可能性があるため、TXブロブ（ダミーアカウントで署名）を`fee`コマンドに送信すると、Xahauは特定のトランザクションに必要な具体的な手数料を返します。

### Fee RPCヘルパー

Hooks Amendmentが有効なレジャー上のトランザクション手数料は、エンドユーザーやウォレットアプリケーションが計算するのに些細ではなくなります。これは、強いHooksがトランザクションの発信者によって支払われなければならず、送信アカウントに最大4つの強いHook、受信アカウントに最大4つの強いHookがある可能性があり、さらに関係する他の強いトランザクション利害関係者（一部の特殊なトランザクションタイプの場合に当てはまる）がいる可能性があるためです。さらに、トランザクションがSetHookである場合、パラメーターのサイズ、コードのサイズ、_create_操作か_install_操作かによって手数料のサイズが決まります。

したがって、**すべての**トランザクションを更新されたfee RPCコールを通して実行してから、レジャーに送信することを強く推奨します。

#### RPCコールを呼び出すには：

1. 作業するHooksノードへのWebSocket接続を開きます。
2. 手数料を知りたいシリアライズされたトランザクションを以下のように作成します：

* `Fee: 0`
* `SigningPubKey: ""`（つまり：型0x73の0バイトVL。16進数で：`0x7300`）
* トランザクションに**署名しない**でください。

3. 次のようにRPCに16進数ブロブとして送信します：

```json
{"command":"fee", "tx_blob":"<hex blob>"}
```

HTTP POST RPCの場合は次のように送信します：

```json
{"method":"fee", "params": [{"tx_blob":"<hex blob>"}] }
```

レスポンスは次のようになるはずです：

```json
{
  result: {
    drops: {
      base_fee: '130520',
    },
  //...
  },
  type: 'response'
}
```

基本手数料を取得し、トランザクションの`Fee`フィールドとして設定します。これで通常のトランザクション送信プロセスに従って署名して送信します。

`tx_blob`の値が無効または`tx_blob`が見つからない場合、通常のJSONレスポンスが`base_fee`が10として返されます。
