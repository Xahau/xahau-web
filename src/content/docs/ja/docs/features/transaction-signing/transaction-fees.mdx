ライブラリが手数料決定を代わりに処理する場合もありますが、Xahau Networkとの独自統合を構築する場合、トランザクション、送信元アカウント、宛先アカウントに基づく動的な手数料決定を実装しなければならないことがあります。

トランザクションの送信者は、特定のトランザクションタイプで起動されるHooksに必要な手数料を支払う必要があります。Hooksは送信元アカウントと宛先アカウントの両方に存在し得ます。TX Blob (ダミーアカウントで署名したもの)を`fee`コマンドへ送信すると、Xahauはその特定のトランザクションに必要な具体的な手数料を返します。

### Fee RPCヘルパー

Hooks Amendmentが有効なレジャー上のトランザクション手数料は、エンドユーザーやウォレットアプリケーションにとって計算が簡単ではありません。これは、strong hooksの費用をトランザクションの発信者が支払う必要があり、送信アカウントに最大4つ、受信アカウントに最大4つのstrong hooksが存在し得るうえ、その他の強いトランザクション上の利害関係者が関与する場合もあるためです (一部の特殊なトランザクションタイプではそのようなことがあります)。さらに、トランザクションがSetHookの場合、パラメーターのサイズ、コードのサイズ、そして _create_ 操作か _install_ 操作かがすべて手数料の大きさを決定します。

そのため、**すべて**のトランザクションはレジャーへ送信する前に、更新済みのfee RPC呼び出しを通すことを強く推奨します。

#### RPC呼び出しを起動するには:

1. 作業対象のHooksノードへwebsocket接続を開きます。
2. 手数料を知りたいシリアライズ済みトランザクションを、以下の内容で構成します。

* `Fee: 0`
* `SigningPubKey: ""` (つまり、タイプ0x73の0バイトVLです。16進数では`0x7300`。)
* トランザクションには署名**しないでください**。

3. 次のようにhex BlobとしてRPCへ送信します。

```json
{"command":"fee", "tx_blob":"<hex blob>"}
```

HTTP POST RPCでは次のように送信します。

```json
{"method":"fee", "params": [{"tx_blob":"<hex blob>"}] }
```

レスポンスは次のようになります。

```json
{
  "result": {
    "drops": {
      "base_fee": "130520",
    },
  //...
  },
  "type": "response"
}
```

base feeを取得し、トランザクションの`Fee`フィールドとして設定します。その後、通常のトランザクション送信プロセスに従って署名し、送信します。

`tx_blob`に無効な値がある場合、または`tx_blob`が欠落している場合は、`base_fee`が10の通常のJSON結果が返されます。
