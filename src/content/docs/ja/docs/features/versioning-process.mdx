このドキュメントは、GitHubリポジトリで使用するバージョニングプロセスを概説しています。`dev`、`candidate`、`release`の3ブランチワークフローを使用します。このプロセスにより、すべての新機能と修正がリリース前に徹底的にテストされます。

### Devブランチ

devブランチは、継続的な開発のためのメインブランチです。すべての新機能、バグ修正、改善がこのブランチにマージされます。これは最もアクティブなブランチであり、開発者はここを作業の基盤とすべきです。

devブランチに貢献するには、次の手順に従います：

1. 自分のGitHubアカウントにリポジトリをフォークします。
2. フォークしたリポジトリをローカルマシンにクローンします。
3. 機能またはバグ修正のための新しいブランチを作成します。
4. 変更を加えてブランチにコミットします。
5. ブランチをGitHub上のフォークリポジトリにプッシュします。
6. ブランチからメインリポジトリのdevブランチへのプルリクエストを作成します。

<Aside type="caution">
devブランチへのすべてのマージはスカッシュ（squash）する必要があります
</Aside>

コードメンテナーがプルリクエストをレビューしてフィードバックを提供します。コードが承認されると、devブランチにマージされます。

### Candidateブランチ

`dev`ブランチの機能がテスト準備完了になると、`candidate`ブランチにマージされます。このブランチは、リリースのほぼ準備が整ったコードのステージングエリアとして機能します。

`candidate`ブランチのコードは徹底的にテストされます。見つかったバグや問題は`dev`ブランチで修正され、`candidate`ブランチにマージされます。

`candidate`ブランチは通常、このテストフェーズに約2週間かかります。ただし、修正の緊急性や新機能の規模によって、この期間は長くなったり短くなったりします。

1. `dev`から新しい`candidate`ブランチを作成します。
2. `candidate`ブランチから`release`ブランチへ「Proposed」のような名前でPRを作成します。
3. 「High Level Overview of Change」を`candidate`のプルリクエストを含むように更新します。「Context of Change」をPRに関する追加メモを含むように更新します。

### Releaseブランチ

`candidate`ブランチのコードが徹底的にテストされ、すべての問題が解決された後、`release`ブランチにマージされます。これがコードリリース前の最終ステップです。

`release`ブランチには、現在本番稼働中または間もなくリリースされるコードが含まれます。完全にテストされた安定したコードのみがこのブランチに入るべきです。

コードが`release`ブランチに入ると、バージョン番号でタグ付けされます。このバージョン番号はリリースの追跡に使用され、リリースノートの作成時にも使用されます。

`release`ブランチでは、https://build.xahau.tech/ でバイナリがビルドされて公開されます。このバイナリがエンドユーザーに提供される最終製品です。

コードをリリースするには、次の手順に従います：

1. `candidate`プルリクエストをreleaseブランチにマージします。
2. `candidate`ブランチを削除します。

<Aside type="caution">
releaseブランチへのすべてのマージは`git merge --ff-only candidate`で行う必要があります
</Aside>

### まとめ

この3ブランチワークフローにより、すべてのコードがリリース前に徹底的にテストされます。本番環境に達する前に問題を発見して修正でき、開発からリリースへのコードの移動に明確なパスを提供します。カスタムLANを使用したテストとプロファイリングメトリクスにより、コードが機能的なだけでなく、効率的でパフォーマンスに優れていることを確保します。
