このドキュメントでは、GitHubリポジトリで使用しているバージョニングプロセスの概要を説明します。`dev`、`candidate`、`release`の3ブランチワークフローを使用します。このプロセスにより、すべての新機能と修正がリリース前に十分にテストされることを確保します。

### Devブランチ

devブランチは継続的な開発のメインブランチです。すべての新機能、バグ修正、改善はこのブランチにマージされます。これは最も活発なブランチであり、開発者はここを作業のベースにする必要があります。

devブランチへ貢献するには、次の手順に従います。

1. リポジトリを自分のGitHubアカウントへフォークします。
2. フォークしたリポジトリをローカルマシンへクローンします。
3. 機能またはバグ修正用の新しいブランチを作成します。
4. 変更を行い、そのブランチへコミットします。
5. ブランチをGitHub上のフォーク済みリポジトリへプッシュします。
6. 自分のブランチからメインリポジトリのdevブランチへpull requestを作成します。

<Aside type="caution">
devブランチへのすべてのマージはsquashする必要があります
</Aside>

コードメンテナーがpull requestをレビューし、フィードバックを提供します。コードが承認されると、devブランチへマージされます。

### Candidateブランチ

`dev`ブランチの機能がテスト可能な状態になったら、`candidate`ブランチへマージします。このブランチは、リリース準備がほぼ整ったコードのステージング領域として機能します。

`candidate`ブランチのコードは十分にテストされます。見つかったバグや問題は`dev`ブランチで修正され、その後`candidate`ブランチへ再度マージされます。

`candidate`ブランチは通常、このテストフェーズに約2週間置かれます。ただし、この期間は修正の緊急度や新機能の規模に応じて長くなったり短くなったりすることがあります。

1. `dev`から新しい`candidate`ブランチを作成します。
2. `candidate`ブランチから`release`ブランチへ、"Proposed"のような名前でPRを作成します。
3. "High Level Overview of Change"を更新して`candidate`からのPull Requestを含めます。"Context of Change"を更新して、PRに関する追加メモを含めます

### Releaseブランチ

`candidate`ブランチのコードが十分にテストされ、すべての問題が対処された後、`release`ブランチへマージされます。これはコードがリリースされる前の最後のステップです。

`release`ブランチには、現在本番環境にあるコード、またはまもなくリリースされるコードが含まれます。このブランチには、十分にテスト済みで安定したコードのみを含める必要があります。

コードが`release`ブランチに入ると、バージョン番号のタグが付けられます。このバージョン番号はリリースの追跡に使用され、リリースノートの作成時にも使用されます。

`release`ブランチでは、バイナリがビルドされ、https://build.xahau.tech/ で公開されます。このバイナリがエンドユーザーに届けられる最終プロダクトです。

コードをリリースするには、次の手順に従います。

1. `candidate` pull requestをreleaseブランチへマージします。
2. `candidate`ブランチを削除します。

<Aside type="caution">
releaseブランチへのすべてのマージは\`git merge --ff-only candidate\`で行う必要があります
</Aside>

### まとめ

この3ブランチワークフローにより、すべてのコードがリリース前に十分にテストされることを確保します。問題が本番環境に到達する前に発見して修正でき、コードを開発からリリースへ進める明確な経路を提供します。テストとプロファイリングメトリクス用にカスタ�LANを使用することで、コードが機能するだけでなく、効率的で高性能であることも確保します。
