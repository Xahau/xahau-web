バリデーターは、提案レジャーごとに署名する追加の公開鍵／秘密鍵ペアを設定した `xahaud` ノードです。そのため、Xahau Network でトランザクション適用順序について合意形成を行う主要な仕組みとなっています。

## 背景と考慮事項
`xahaud` をバリデーターとして運用するのは一見簡単そうに見えますが、本番ノードの運用には可用性やセキュリティなど多くの考慮点があります。以下の導入セクションでは、それらの背景を説明します。

### セキュリティと可用性
バリデーターとして `xahaud` を運用する場合、システムのセキュリティと可用性は極めて重要です。バリデーター運用者には、サーバーを継続的に監視し、障害時に迅速に対応することが求められます。多様なデータセンターに待機系マシンを用意し、障害復旧を素早く行えるようにしている運用者もいます。自宅で信頼性の高いバリデーターを運用すること自体は可能ですが、その構成で広範な信頼を獲得し、たとえば公開 UNL に掲載される水準の安定性や安全性を提供するのは難しいでしょう。同様に、`xahaud` バリデーターとして使うマシンで `xahaud` 以外のソフトウェアを動かすことは強く非推奨です。リソース不足や脆弱性露出の可能性が高まるためです。WebSocket や RPC アクセスが必要な管理者は、照会専用の別マシンを使い、レジャー検証という重要な処理からリソースを奪わないようにすることが推奨されます。
<br></br>
秘密／公開のバリデーション鍵ペアは、安全なオフライン環境に保管し、バックアップしてください。信頼されたバリデーターの鍵にアクセスできる者は、ネットワークへ損害を与える可能性があります。`xahaud.cfg` にバリデーショントークンを設定した後は、読み取り権限を制限し（`chmod 400 xahaud.cfg`）、インストール形態に応じてファイル所有者を `root` または `xahaud` に変更する（`chown root:root xahaud.cfg`）ことが推奨されます。

### アイデンティティ証明
[Sybil attack](https://en.wikipedia.org/wiki/Sybil_attack)を避け、信頼を構築するために、検証を行う `xahaud` ノードは自身のドメイン名をバリデーション鍵で暗号学的に署名してアイデンティティを証明します。この署名は同じドメイン上で [TOML 形式ファイル](/docs/infrastructure/identity)として公開されます。バリデーターの身元確認に加え、この TOML ファイルは Xahau Network 上の個別アカウント保有者の証明にも利用できます。その場合は秘密バリデーション鍵ではなく秘密アカウント鍵で署名します。TOML ファイルは、TLS を有効にした Web サーバー上の以下の場所で公開する必要があります。<br></br><code>https://[your-domain.com]/.well-known/xahau.toml</code><br></br>

### ピアリング
高いセキュリティを維持するため、バリデーターとして構成された `xahaud` ノードは、既知かつ信頼できるピアとのみ接続するべきです。`xahaud.cfg` で `peer_private` を有効にし、`ips_fixed` を明示的に定義すると、サーバーは定義されたピアにのみ接続するようになります。ファイアウォールなど OS やハードウェアレベルの保護も、定義済みピアのみに接続しているという安心材料になります。ただし、ピア数が少なすぎる、一般的には 5 未満のバリデーターは、ネットワーク全体との同期を失うリスクがあります。

### ガバナンスと信頼
Xahau Network は、ネットワーク管理の民主的手法として [Governance Game](/docs/features/governance-game) を採用しています。さらに、ネットワークに参加する各 `xahaud` インスタンスは、Unique Node List（UNL）と呼ばれる信頼済みバリデーター一覧を設定する必要があります。そのため、既存のガバナンス参加者は新しいバリデーターをガバナンステーブルに迎え入れる投票ができます。別途、[InFTF](https://inftf.org) のような信頼された UNL 発行者が、推奨信頼バリデーター一覧へ含めることもできます。バリデーターを運用しても、Governance Game や UNL への掲載が保証されるわけではありません。通常、ガバナンス過程へ組み込まれる検証サーバーには次の特徴があります。
1. 高可用性
    - 24 時間 365 日の有効な監視と、障害への迅速な対応
2. ネットワーク全体との高い合意率
3. 所有者／運営者が特定されており、地理的位置が明確であること
4. [Amendments](/docs/features/amendments) や手数料の更新・投票に積極的であること
    - これには開発状況を継続的に確認する時間が必要です

## バリデーションを有効にする
標準の `xahaud` サーバーをバリデーターへ切り替える手順は比較的簡単です。基本的には公開鍵／秘密鍵ペアを生成し、そのマスター鍵ペアから導かれる一時鍵でトークンを作成します。そのトークンを `xahaud.cfg` に設定することで、`xahaud` がネットワークへ検証提案を送るようになります。

### バリデーション鍵ツールのビルド
現時点では、バリデーション鍵生成に使うソフトウェアの公式バイナリ配布はありません。そのため、ユーザーは自分でツールをビルドするか、非公式ビルドを利用する必要があります。ビルド手順は [xahaud のビルド](../build-xahaud) と同じで、[Xahau/validator-keys-tool](https://github.com/Xahau/validator-keys-tool) リポジトリにも案内があります。また、非公式リポジトリにはいくつかの[パッケージ済みバイナリ](https://github.com/jscottbranson/xahau-examples/)もあります。本番バリデーター用の鍵を生成する場合は、安全なエアギャップ環境でソースからビルドするべきです。

`xahaud` は `validation_create` [Admin API メソッド](/docs/features/http-websocket-apis/admin-api-methods)を提供していますが、このメソッドは必要なバリデーショントークンを生成したり、ドメイン検証を可能にしたりはできません。そのため名前に反して、`validation_create` はサーバーを一般的に識別するための公開鍵／秘密鍵を作る用途、たとえば[複数サーバーのクラスタリング](/docs/infrastructure/advanced-configuration)に使われます。

### バリデーション鍵の生成とインストール
バリデーション鍵とバリデーショントークンの生成には、オフラインマシンの利用を検討してください。
1. バリデーション鍵ツールのあるディレクトリへ移動します: `cd [/path/to/tool]`
2. 新しい鍵ペアを生成します: `./validator-keys create_keys`
3. 生成された鍵ペアは `/home/[username]/[.ripple or .xahaud]/validator-keys.json` に保存されます。保存先は validator-keys-tool のバージョンによって異なります。
4. 理想的には、バリデーターはアイデンティティ証明を提供するべきです。そのため、validator-keys.json にドメインを設定します: `./validator-keys set_domain [example.com]`
    - 識別用ドメインを公開したくない場合は、ドメインなしでバリデーショントークンを生成できます: `./validator-keys create_token --keyfile /home/[username]/.xahaud/validator-keys.json`
5. 直前のコマンド出力を確認します。`[validator_token]` は `xahaud.cfg` に設定してバリデーションを有効化するため、控えてください。ドメインを設定した場合は `attestation=""` 行も控えておき、ドメイン上で公開する TOML ファイルへ追加する必要があります。
    - このツールで新しい `validator_token` を生成するたび、`validator-keys.json` 内の `token_sequence` が増加します。ネットワークは、観測済み最大番号より低いシーケンス番号で生成されたトークンからの検証を無視します。たとえば現在のトークンがシーケンス番号 `3` で、新たに `2` のトークンを生成しても、その低いシーケンス番号を使うマシンからの検証は無視されます。この仕組みにより、セキュリティ侵害などが発生した際に、マスター検証鍵を再生成せず新しいトークンだけを発行できます。
6. `xahaud.cfg` を開いて `[validation_token]` を貼り付け、`xahaud` サービスを再起動します。
7. バリデーターの身元を証明するため、`attestation=""` を自身の [TOML](/docs/infrastructure/identity) ファイルへ追加します。
8. 秘密バリデーション鍵は安全に保管し、可能ならバリデーター本体とは別の場所へ置いてください。
