validatorは、提案された各レジャーに署名するために使われる追加の公開/秘密鍵ペアを構成したxahaudノードです。そのため、validatorはXahau Networkでトランザクションの適用順序に関するコンセンサスを達成するための主要な仕組みです。

## 背景の考慮事項
validationを行うxahaudサーバーの運用は簡単に見えるかもしれませんが、本番ノード運用者には稼働率やセキュリティなど、考慮すべき点が数多くあります。以下の導入セクションでは、これらの考慮事項の背景を説明します。

### セキュリティと可用性
xahaudをvalidatorとして運用する場合、システムのセキュリティと可用性は極めて重要です。validator運用者はサーバーを継続的に監視し、障害に迅速に対応することが期待されます。いくつかのvalidator運用者は、障害からすばやく復旧できるよう、複数のデータセンターに待機機を置くことを選びます。自宅でも信頼性のあるvalidatorを運用することは可能ですが、そのような構成がvalidatorに広範な信頼 (たとえば公開UNLへの掲載)を得られるだけの安定性とセキュリティを提供する可能性は低いでしょう。同様に、xahaud validatorとして使うマシンではxahaud以外のソフトウェアを実行しないことを強く推奨します。そうしないと、リソース不足やセキュリティ脆弱性の露出の可能性が高まるからです。サーバーへのWebSocketまたはRPCアクセスを必要とするシステム管理者は、問い合わせ用に別のマシンを使うことを推奨します。そうすることで、レジャー検証という重要な作業からリソースが奪われません。
<br></br>
private/public validation keypairは、安全なオフライン環境に保存し、バックアップしてください。trusted validatorの鍵にアクセスできる人は、ネットワークに害を与える可能性があります。validation tokenを `xahaud.cfg` にインストールした後は、読み取り権限を制限し (`chmod 400 xahaud.cfg`)、インストール形態に応じてファイル所有者を `root` または `xahaud` に変更することを推奨します (`chown root:root xahaud.cfg`)。

### ID attestation
[Sybil attack](https://en.wikipedia.org/wiki/Sybil_attack)を避けて信頼を構築するために、validationを行うxahaudノードはvalidation keyでドメイン名に暗号学的署名を行い、自分のIDを証明します。この署名は同じドメイン上の [TOML形式ファイル](/ja/docs/infrastructure/identity)にホストされます。このTOMLファイルは、validatorのIDを検証するだけでなく、private validation keyの代わりにprivate account keyで署名したattestationを提供することで、Xahau Network上の個々のアカウント保有者のIDも検証できます。TOMLファイルはTLS暗号化を有効にしたWebサーバーで次の場所にホストする必要があります:<br></br><code>https://[your-domain.com]/.well-known/xahau.toml</code><br></br>

### ピアリング
高いセキュリティを維持するため、validatorとして構成されたxahaudノードは、既知で信頼できるピアとのみピアリングすべきです。`peer_private` 設定を有効にし、`xahaud.cfg` で `ips_fixed` を明示的に定義すると、サーバーは定義済みピアにのみ接続するようになります。ファイアウォールなどのOSおよびハードウェアレベルの保護は、サーバーが定義済みピアにのみ接続しているという追加の安心材料になります。ただし、ピアが少なすぎるvalidatorは、通常5台未満の場合、ネットワークの他の部分と同期を失うリスクがあるため、これは難しいことがあります。

### ガバナンスと信頼
Xahau Networkは、Networkを民主的に管理する方法として [Governance Game](/ja/docs/features/governance-game)を利用しています。さらに、Networkに参加する各xahaudインスタンスは、Unique Node List (UNL)と呼ばれる信頼済みvalidationサーバーの一覧をインストールする必要があります。そのため、既存のガバナンス参加者は新しいvalidatorをガバナンスの席に迎えるために投票できます。別途、[InFTF](https://inftf.org)のような信頼済みUNL発行者は、validatorを推奨するtrusted validatorsの一覧に含めることを選べます。validatorを運用しても、Governance Gameまたは任意のUNLへの掲載が保証されるわけではありません。通常、ガバナンスプロセスへの掲載対象として選ばれるvalidationサーバーには、次のような特徴があります。
1. 高い可用性
    -障害に迅速に対応する24/7/365の効果的な監視
2. ネットワークの他の部分との高い一致率
3. 所有者/運用者が明確で、所在地もはっきりしている
4. [amendments](/ja/docs/features/amendments)と手数料の更新および投票に積極的に取り組む
    -これには、開発中の内容を追い続ける時間を確保する必要があります

## validationを有効にする
標準のxahaudサーバーをvalidatorに切り替えるのは、比較的簡単な手順です。基本的には、ユーザーが公開/秘密鍵ペアを生成し、そのmaster pairから派生した一時鍵を使ってtokenを生成します。そのtokenを `xahaud.cfg` にインストールすることで、xahaudがNetworkにvalidationを提案するようになります。

### Validation Keys Toolのビルド
現時点では、validation keyを生成するソフトウェアの公式バイナリリリースはありません。そのため、ユーザーはツールをゼロからビルドするか、非公式ビルドを利用することになります。ツールのビルド手順は [xahaudのビルド](../build-xahaud)の手順と同じで、[Xahau/validator-keys-tool](https://github.com/Xahau/validator-keys-tool)リポジトリページに記載されています。非公式リポジトリには [パッケージ済みバイナリ](https://github.com/jscottbranson/xahau-examples/)もあります。本番validator用の鍵を生成したいユーザーは、安全なエアギャップ環境のマシンでツールをゼロからビルドするべきです。

xahaudには `validation_create` [Admin APIメソッド](/ja/docs/features/http-websocket-apis/admin-api-methods)がありますが、このメソッドでは必要なvalidation tokenを生成したり、ドメイン検証を行ったりすることはできません。そのため、名前に反して `validation_create` メソッドは、たとえば [複数サーバーをクラスタリングする](/ja/docs/infrastructure/advanced-configuration)場合のように、サーバーを広く識別するための公開/秘密鍵を作成するために使います。

### validation keyの生成とインストール
validation keyとvalidation tokenの生成には、オフラインマシンを使うことを検討してください。
1. validation keys toolがあるディレクトリへ移動します: `cd [/path/to/tool]`
2. 新しいkeypairを生成します: `./validator-keys create_keys`
3. 新しく生成されたkeypairは `/home/[username]/[.ripple or .xahaud]/validator-keys.json` に保存されます。保存先は使用するvalidator-keys-toolのバージョンによって異なります。
4. 理想的には、validatorはID attestationを提供すべきです。そのためには、validator-keys.jsonファイルでドメインを設定します: `./validator-keys set_domain [example.com]`
    -識別用ドメインを提供したくない場合は、ドメインなしでvalidation tokenを生成します: `./validator-keys create_token --keyfile /home/[username]/.xahaud/validator-keys.json`
5. 直前のコマンドの出力を確認します。`[validator_token]` を控えてください。これはvalidationを有効にするためにxahaud.cfgにインストールします。ドメインを設定した場合は、`attestation=""` 行も控えてください。これはドメインで配信するTOMLファイルに追加する必要があります。
    -ツールで新しい `validator_token` を生成するたびに、`token_sequence` フィールドがvalidator-keys.jsonファイル内で増加します。Networkは、観測済み最大値より小さいsequence numberで生成されたtokenを持つサーバーからのvalidationを無視します。たとえば、validation tokenのsequence numberが "3" で、新しいtokenをsequence number "2" で生成した場合、新しいtokenを使うマシンからのvalidationは無視されます。この機能により、セキュリティ侵害やその他のインシデントが発生した場合に、新しいmaster validation keyを生成するのではなく、新しいtokenを生成できます。
6. xahaud.cfgを開いて `[validation_token]` を貼り付け、その後xahaudサービスを再起動します。
7. validatorのIDを証明するため、`attestation=""` を [TOML](/ja/docs/infrastructure/identity)ファイルに貼り付けます。
8. private validation keyは、安全に、理想的にはvalidatorとは別に保管してください。
