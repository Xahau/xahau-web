---
title: システム要件
---
さまざまな用途向けにカスタマイズされたノードのハードウェア要件 (RPC/WS、ハブ、バリデーターなど)は、以下の仕様と一致している必要があります。想定するユースケースに応じて、各システムを調整してください。

## ノードの種類
ユーザーのニーズに応じて、xahaudは多様な構成で実行できます。同様に、1つのxahaudサーバーを、RPC/WS呼び出しにも使うフルヒストリーサーバーのように、複数の目的向けに構成することもできます。サーバーの用途が必要なシステム要件を決定します。xahaudの構成には次のようなものがあります。
1. 標準サーバー- validatorとして構成されていない任意のxahaudノードを指すために一般的に使われる用語です。標準サーバーは通常、既定の構成からの変更がごくわずかです。
2. フルヒストリーサーバー-過去データの照会に役立ちます。大量のストレージ容量に加えて、サーバーを照会するユーザー数に応じて、より高いディスクIOPSとネットワーク帯域幅が必要になる場合があります。
3. RPC/WebSocketサーバー-トランザクションの送信や、オーダーブックや検証ストリームなどの現在のレジャーデータの照会に適しています。NginxなどのプロキシソフトウェアをRPC/WSサーバーの前段に置くことで、負荷分散、暗号化、その他の利点を提供することがよくあります。
4. ハブサーバー-ノード間のピアリングはXahau Network全体でメッセージを中継するうえで重要であり、1台のサーバーで数百のピアへのピアリングを提供するよう構成できます。ただし、非常に大きな帯域幅 (10 Gbit以上)が必要になる場合があります。
5. [検証サーバー](/ja/docs/infrastructure/enabling-validation) -バリデーターは、提案されたレジャー内のトランザクション列に署名できるよう、追加の暗号鍵をインストールする必要がある点で他の種類のノードとは異なります。可用性とセキュリティが重要であるため、バリデーターを他の用途に使うことは推奨されません。

## 一般的な注意
* ビルドは通常Ubuntu LTSでテストされています。ただし、Docker、ポータブルバイナリ、カスタムビルド、その他のインストール方法により、さまざまなオペレーティングシステムをサポートできます。
* 通常、xahaudインスタンスごとに1つのIPv4またはIPv6アドレスが必要です。1つのIPv4アドレスの背後で複数のインスタンスを実行することは推奨されません。
* フルヒストリーを保存するマシンでは、EXT4の単一ファイルサイズ制限 (最大16TB)を避けるため、XFSか同等のファイルシステムを使用する必要があります。保存する履歴が少ない場合はEXT4で十分です。
* 下記の本番サーバー向け「最小」仕様を満たさないマシンでもxahaudを実行できる場合があります。ただし、その場合は不安定になるリスクがあります。
* これらのシステム要件は、時間の経過とともに増加する可能性があります。たとえば、フルヒストリーサーバー向けのディスク容量は継続的に増えています。
* 2025年11月1日時点で、<strong>Xahau Networkのフルヒストリーは10TBです</strong>。
