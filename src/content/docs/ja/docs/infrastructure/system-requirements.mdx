---
title: システム要件
---
RPC / WS、ハブ、バリデーターなど、さまざまな用途向けに構成されたノードのハードウェア要件は、以下の仕様を目安にしてください。実際の用途に応じて個別に調整することをおすすめします。

## ノードの種類
`xahaud` は、利用者のニーズに応じてさまざまな構成で実行できます。同様に、1 台の `xahaud` サーバーを複数用途、たとえばフルヒストリーサーバー兼 RPC / WS サーバーとして構成することも可能です。必要なシステム要件はサーバーの目的によって決まります。代表的な構成には次のようなものがあります。
1. 標準サーバー - 一般に、バリデーターとして設定されていない任意の `xahaud` ノードを指します。標準サーバーは通常、デフォルト設定からの変更がほとんどありません。
2. フルヒストリーサーバー - 過去データの照会に有用です。大量のストレージに加え、サーバーを照会する利用者数に応じて、より高いディスク IOPS やネットワーク帯域が必要になる場合があります。
3. RPC / WebSocket サーバー - トランザクション送信や、オーダーブック、バリデーションストリームのような最新レジャーデータの照会向けに最適化されます。ロードバランシング、暗号化、その他の利点を得るため、Nginx などのプロキシソフトウェアを前段に置くことが一般的です。
4. ハブサーバー - ノード間のピアリングは Xahau Network 全体へメッセージを中継するうえで重要であり、1 台のサーバーで数百ピア向けにピアリングを提供する構成も可能です。ただし、大きな帯域幅（10 Gbit 以上）が必要になることがあります。
5. [バリデーションサーバー](/docs/infrastructure/enabling-validation) - バリデーターは、提案レジャー内のトランザクション順序へ署名するための追加暗号鍵が必要な点で他ノードと異なります。可用性とセキュリティが極めて重要なため、他の用途と兼用することは推奨されません。

## 一般的な注意事項
* ビルド検証は通常 Ubuntu LTS で行われます。ただし、Docker、ポータブルバイナリ、カスタムビルド、その他のインストール方法により、多様な OS をサポートできます。
* 通常、各 `xahaud` インスタンスには 1 つの IPv4 および / または IPv6 アドレスが必要です。1 つの IPv4 アドレスの背後で複数インスタンスを運用することは推奨されません。
* フルヒストリーを保存するマシンでは、EXT4 の単一ファイルサイズ制限（最大 16TB）を避けるために XFS または同等のファイルシステムを使う必要があります。保存履歴が少ないなら EXT4 でも十分です。
* 下記「Minimum」を満たさないマシンでも `xahaud` を動かせる場合はありますが、本番環境では不安定化のリスクがあります。
* これらのシステム要件は時間とともに増加する可能性があります。たとえば、フルヒストリーサーバーに必要なディスク容量は継続的に増えています。
* 2025 年 11 月 1 日時点で、<strong>Xahau Network のフルヒストリー容量は 10TB</strong> です。
