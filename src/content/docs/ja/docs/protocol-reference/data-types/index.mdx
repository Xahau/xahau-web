オブジェクトの種類は、それぞれ異なる方法で一意に識別されます。

アカウントはAddressによって識別されます。たとえば `"r9cZA1mLK5R5Am25ArfXFmqgNwjZgnfk59"` です。Addressは常に "r" で始まります。多くの `rippled` メソッドは16進表現も受け付けます。

トランザクションは、そのバイナリ形式のHashによって識別されます。送信元アカウントとSequence Numberでもトランザクションを識別できます。

各クローズ済みLedgerには、Ledger IndexとHash値があります。\[Specifying Ledgers]\[]では、どちらも使用できます。

### ハッシュ

#### ハッシュプレフィックス

[\[出典\]](https://github.com/Xahau/xahaud/blob/master/src/ripple/protocol/HashPrefix.h)

多くの場合、Xahauはハッシュを計算する前に、オブジェクトのバイナリデータの先頭に4バイトのコードを付けて、バイナリ形式が同じでも異なる種類のオブジェクトが異なるハッシュになるようにします。既存の4バイトコードは、ASCIIでエンコードされた3つの英字の後にゼロバイトを続ける構造です。

ハッシュの種類の中には、APIリクエストやレスポンスに現れるものもあります。ほかのものは、ある種類のデータに署名する最初のステップとして、またはより上位のハッシュを計算するためだけに使われます。次の表は、Xahauが使用するすべての4バイトのハッシュプレフィックスを示しています。

| オブジェクト型                          | APIフィールド                   | ハッシュプレフィックス（Hex） | ハッシュプレフィックス（Text） |
| ------------------------------------- | -------------------------------- | ---------------------------- | ----------------------------- |
| コンセンサス提案                      | N/A                              | `0x50525000`                 | `PRP\0`                       |
| レジャーバージョン                    | `ledger_hash`                    | `0x4C575200`                 | `LWR\0`                       |
| レジャー状態データ                    | レジャーヘッダーの `account_state` | `0x4D4C4E00`                 | `MLN\0`                       |
| レジャーデータの内部ノード            | N/A                              | `0x4D494E00`                 | `MIN\0`                       |
| レジャーデータの内部ノード (SHAMapv2) | N/A                              | `0x494E5200`                 | `INR\0`                       |
| Payment Channel Claim                 | N/A                              | `0x434C4D00`                 | `CLM\0`                       |
| 署名済みトランザクション              | トランザクションの `hash`         | `0x54584E00`                 | `TXN\0`                       |
| メタデータ付きトランザクション        | N/A                              | `0x534E4400`                 | `SND\0`                       |
| 未署名トランザクション（単一署名）    | N/A                              | `0x53545800`                 | `STX\0`                       |
| 未署名トランザクション（マルチ署名）  | N/A                              | `0x534D5400`                 | `SMT\0`                       |
| 検証投票                              | N/A                              | `0x56414C00`                 | `VAL\0`                       |
| バリデーターマニフェスト              | N/A                              | `0x4D414E00`                 | `MAN\0`                       |

レジャーオブジェクトIDも同様の方法で計算されますが、ここで説明した形式のプレフィックスではなく、「スペースキー」と呼ばれる2バイトのプレフィックスを使用します。

### レジャーインデックス

#### レジャーの指定

多くのAPIメソッドでは、取得したデータが共有レジャーのその特定バージョン時点で最新であるとみなされるように、レジャーのインスタンスを指定する必要があります。レジャーバージョンを受け付けるコマンドは、どれも同じ方法で動作します。使用したいレジャーを指定する方法は3つあります。

1.  `ledger_index` パラメーターでレジャーインデックスによってレジャーを指定します。各クローズ済みレジャーのレジャーインデックスは、直前のレジャーより1ずつ大きくなります。（最初のレジャーのレジャーインデックスは1でした。）

    ```
     "ledger_index": 61546724
    ```
2.  `ledger_hash` パラメーターでハッシュ値によってレジャーを指定します。

    ```
     "ledger_hash": "8BB204CE37CFA7A021A16B5F6143400831C4D1779E6FE538D9AC561ABBF4A929"
    ```
3. `ledger_index` パラメーターで、次のショートカットのいずれかを使ってレジャーを指定します。
   *   `validated` は、コンセンサスによって検証された最新のレジャーを表します。

       ```
         "ledger_index": "validated"
       ```
   * `closed` は、変更受付を終了し、検証のために提案された最新のレジャーを表します。
   * `current` は、サーバーの現在の作業中のレジャーバージョンを表します。

上記3形式のいずれも受け付ける、非推奨の `ledger` パラメーターもあります。このパラメーターは_使用しないでください_。予告なく削除される可能性があります。

レジャーを指定しない場合、サーバーがリクエストの応答に使うレジャーを決定します。既定では、サーバーは `current`（進行中）のレジャーを選びます。Reporting Modeでは、代わりに最新の検証済みレジャーを使います。レジャーを指定するフィールドは1つだけにしてください。

**注:** レジャーの指定における既定動作に依存しないでください。これは変更される可能性があります。可能であれば、常にリクエストでレジャーバージョンを指定してください。

Reporting Modeは、検証されるまでレジャーデータを記録しません。Reporting Modeサーバーに `current` または `closed` のレジャーをリクエストすると、そのサーバーはリクエストをP2P Modeサーバーに転送します。検証されていないレジャーインデックスまたはハッシュを要求した場合、Reporting Modeサーバーは `lgrNotFound` エラーを返します。

### 通貨金額の指定

Xahauには2種類の通貨があります。XAHとトークンです。これら2種類の通貨は、それぞれ異なるフォーマットで指定され、精度や丸めの挙動も異なります。

送信先 `Amount` のように、\[Payment transaction]\[]の一部フィールドはどちらの型でも指定できます。一方で、`Fee` フィールド（トランザクションコスト）のように、XAHのみを受け付けるフィールドもあります。

XAHは、XAHの "drops" の整数を含む文字列として指定します。100万dropsが1 XAHに相当します。トークンは、10進の金額、通貨コード、発行者のフィールドを持つオブジェクトとして指定します。たとえば次のようになります。

*   **XAH** - 13.1 XAHの値を持つ `Amount` フィールドを指定するには、次のようにします。

    ```
      "Amount": "13100000"
    ```
*   **Token** - `rf1B...` が発行した、または `rf1B...` 宛ての13.1 FOOの値を持つ `Amount` フィールドを指定するには、次のようにします。

    ```
      "Amount": {
          "value": "13.1",
          "currency": "FOO",
          "issuer": "rf1BiGeXwwQoi8Z2ueFYTEXSwuJYfV2Jpn"
      }
    ```

詳細は、通貨フォーマットを参照してください。

### 時刻の指定

`xahaud` サーバーとそのAPIでは、時刻は符号なし整数として表されます。この数値は、2000年1月1日00:00 UTCの "Ripple Epoch" からの秒数を表します。これは [Unix epoch](http://en.wikipedia.org/wiki/Unix_time)と似ていますが、Ripple EpochはUnix Epochより946684800秒後です。

Ripple Epochの時刻を32ビット変数でUNIX Epochの時刻に変換しないでください。整数オーバーフローの原因になる可能性があります。
