---
title: トランザクション共通フィールド
---

すべてのトランザクションには同じ共通フィールドのセットがあり、さらにトランザクションタイプに応じた追加フィールドがあります。フィールド名は大文字小文字を区別します。すべてのトランザクションに共通するフィールドは次のとおりです:

| フィールド            | JSON型           | \[内部型]\[] | 説明                                                                                                                                                                                                                                                                                                                                |
| -------------------- | ---------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `Account`            | String           | AccountID           | _(必須)_ トランザクションを開始したアカウントの一意のアドレス。                                                                                                                                                                                                                                                                         |
| `TransactionType`    | String           | UInt16              | _(必須)_ トランザクションの種類。有効なトランザクションタイプには、`Payment`、`OfferCreate`、`TrustSet` などがあります。                                                                                                                                                                                                                  |
| `Fee`                | String           | Amount              | _(必須; 自動補完可)_ このトランザクションをネットワークに配布するためのコストとして破棄される、drops単位のXAHの整数金額。一部のトランザクションタイプには異なる最小要件があります。詳細は \[トランザクションコスト]\[]を参照してください。                                                                                               |
| `Sequence`           | Number           | UInt32              | _(必須; 自動補完可)_ トランザクションを送信するアカウントのシーケンス番号。トランザクションは、`Sequence` 番号が同じアカウントからの直前のトランザクションよりちょうど1大きい場合にのみ有効です。特例として `0` は、代わりにTicketを使用していることを意味します _(\[TicketBatch Amendment]\[]によって追加。)_. |
| `AccountTxnID`       | String           | Hash256             | _(任意)_ 別のトランザクションを識別するハッシュ値。指定した場合、このトランザクションは送信元アカウントの以前に送信されたトランザクションが指定されたハッシュと一致するときにのみ有効です。                                                                                                                                       |
| `Flags`              | Number           | UInt32              | _(任意)_ このトランザクションのビットフラグのセット。                                                                                                                                                                                                                                                                                     |
| `LastLedgerSequence` | Number           | UInt32              | _(任意; 強く推奨)_ このトランザクションが出現できる最も大きいレジャーインデックス。これを指定すると、トランザクションが検証または拒否されるまで待機できる時間の厳格な上限が設定されます。詳細はReliable Transaction Submissionを参照してください。                                                                          |
| `Memos`              | Array of Objects | Array               | _(任意)_ このトランザクションを識別するために使われる追加の任意情報。                                                                                                                                                                                                                                                                  |
| `NetworkID`          | Number           | UInt32              | _(ネットワーク固有)_ このトランザクションの対象となるチェーンのネットワークID。Mainnetと一部のテストネットワークでは **省略する必要があります**。ネットワークIDが1025以上のチェーンでは **必須** です。                                                                                                                    |
| `Signers`            | Array            | Array               | _(任意)_ このトランザクションを承認するマルチ署名を表すオブジェクトの配列。                                                                                                                                                                                                                                                              |
| `SourceTag`          | Number           | UInt32              | _(任意)_ この支払いの理由、またはこのトランザクションが代理で行われる送信者を識別するための任意の整数。慣例として、返金は元の支払いの `SourceTag` を返金支払いの `DestinationTag` として指定します。                                                                                                                         |
| `SigningPubKey`      | String           | Blob                | _(署名時に自動追加)_ このトランザクションに署名するために使われた秘密鍵に対応する公開鍵の16進表現。空文字列の場合は、代わりに `Signers` フィールドにマルチ署名が存在することを示します。                                                                                                    |
| `TicketSequence`     | Number           | UInt32              | _(任意)_ `Sequence` 番号の代わりに使用するTicketのシーケンス番号。これが指定される場合、`Sequence` は `0` でなければなりません。`AccountTxnID` と併用できません。                                                                                                                                                                    |
| `TxnSignature`       | String           | Blob                | _(署名時に自動追加)_ このトランザクションが、記載されているアカウントから発行されたものであることを検証する署名。                                                                                                                                                                                                                      |
| `EmitDetails`        | Object           | Object              | 発行に関する詳細を含みます。これには、発行の世代、発行の負荷、コールバックアドレス、トランザクションを発行したHookのハッシュ、発行のnonce、および親トランザクションのIDが含まれます。                                                                                   |
| `HookParameters`     | Array            | Array               | txのHookパラメーター。                                                                                                                                                                                                                                                                                                             |

\[rippled 0.28.0で削除]\[]:トランザクションの `PreviousTxnID` フィールドは `AccountTxnID` フィールドに置き換えられました。このString / Hash256フィールドは一部の履歴トランザクションに存在します。これは、一部のレジャーオブジェクトにある同名の `PreviousTxnID` フィールドとは関係ありません。

### AccountTxnID

`AccountTxnID` フィールドを使うと、トランザクションを連結できるため、同じアカウントから送信された直前のトランザクションが特定の \[transaction hash]\[識別ハッシュ]と一致しない限り、現在のトランザクションは無効になります。

`PreviousTxnID` フィールドは、送信者に関係なくアカウントを _変更した_ 最後のトランザクションを追跡しますが、`AccountTxnID` はアカウントによって _送信された_ 最後のトランザクションを追跡します。`AccountTxnID` を使用するには、まず `asfAccountTxnID` フラグを有効にして、レジャーがそのアカウントの直前のトランザクションIDを追跡するようにする必要があります。(`PreviousTxnID` は、比較すると常に追跡されます。)

これが役立つ状況の1つは、トランザクションを送信するためのメインシステムと、待機系のバックアップシステムがある場合です。待機系バックアップがメインシステムから切断されたが、メインシステムが完全には停止しておらず、両方が同時に動作し始めると、一部のトランザクションが二重送信され、他がまったく送信されないなど、深刻な問題が発生する可能性があります。`AccountTxnID` でトランザクションを連結しておけば、両方のシステムが稼働していても、一度に有効なトランザクションを送信できるのはどちらか一方だけになります。

`AccountTxnID` フィールドは、Ticketsを使用するトランザクションでは使用できません。`AccountTxnID` を使用するトランザクションは、トランザクションキューに入れることができません。

### 自動補完可能フィールド

一部のフィールドは、`xahaud` サーバーまたはクライアントライブラリによって、トランザクションに署名する前に自動的に埋めることができます。値の自動補完には、最新の状態を取得するためにXahauへのアクティブな接続が必要なため、オフラインでは実行できません。詳細はライブラリによって異なりますが、自動補完は少なくとも次のフィールドに適切な値を常に提供します:

*   `Fee` -ネットワークに基づいて \[トランザクションコスト]\[]を自動的に補完します。

    **注:** `xahaud` の \[signコマンド]\[]を使う場合は、`fee_mult_max` と `fee_mult_div` パラメーターを使って、自動補完されうる値の最大値を制限できます。)
* `Sequence` -トランザクションを送信するアカウントの次のシーケンス番号を自動的に使用します。

本番システムでは、これらのフィールドをサーバーに補完させたままにしないことを推奨します。たとえば、ネットワーク負荷の一時的な急増によってトランザクションコストが高くなった場合、トランザクションの一部を送信する前にコストが下がるのを待つほうが、一時的に高いコストを支払うより望ましいことがあります。

\[Payment transaction]\[]型の `Paths` フィールドも、自動的に補完できます。

### Flagsフィールド

`Flags` フィールドには、トランザクションの動作に影響するさまざまなオプションを含めることができます。オプションはビット値として表され、ビット演算のORを使って複数のフラグを同時に設定できます。

トランザクションに特定のフラグが有効かどうかを確認するには、フラグの値と `Flags` フィールドに対してビット演算のANDを使用します。結果が0の場合はフラグが無効、フラグ値と等しい場合はフラグが有効です。（それ以外の結果になった場合は、何かが間違っています。）

ほとんどのフラグは、特定のトランザクションタイプに対してのみ意味を持ちます。同じビット値が異なるトランザクションタイプのフラグに再利用される場合があるため、フラグの設定や読み取りでは `TransactionType` フィールドに注意を払うことが重要です。

フラグとして定義されていないビットは、必ず0でなければなりません。（\[fix1543 Amendment]\[]は、一部のトランザクションタイプでこのルールを強制します。ほとんどのトランザクションタイプでは、既定でこのルールが強制されます。）

#### グローバルフラグ

すべてのトランザクションにグローバルに適用されるフラグは次の1つだけです:

| フラグ名              | 16進値        | 10進値        | 説明                                                                                                                                 |
| --------------------- | ------------ | ------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `tfFullyCanonicalSig` | `0x80000000` | 2147483648    | **非推奨** 効果はありません。（\[RequireFullyCanonicalSig Amendment]\[]が有効でない場合、このフラグは完全正規署名を強制します。） |

\[signメソッド]\[]（または "sign-and-submit" モードの \[submitメソッド]\[]）を使用すると、`xahaud` は `Flags` フィールドがまだ存在しない限り、`tfFullyCanonicalSig` を有効にした `Flags` フィールドを追加します。`Flags` が明示的に指定されている場合、`tfFullyCanonicalSig` フラグは自動的には有効になりません。マルチ署名済みトランザクションに署名を追加するために \[sign_forメソッド]\[]を使用しても、このフラグは自動的には有効になりません。

**注:** `tfFullyCanonicalSig` フラグは、レガシーな署名ソフトウェアとの互換性を維持しながらトランザクションの可変性を防ぐために、2014年から2020年まで使用されていました。\[RequireFullyCanonicalSig Amendment]\[]により、そのようなレガシーソフトウェアとの互換性は終了し、保護はすべてのトランザクションの既定になりました。RequireFullyCanonicalSigが有効でない並列ネットワークを使用している場合は、トランザクションの可変性から保護するために、常に `tfFullyCanonicalSig` フラグを有効にしてください。

#### フラグの範囲

トランザクションの `Flags` フィールドには、異なるレベルやコンテキストに適用されるフラグを含めることができます。各コンテキストのフラグは、次の範囲に制限されます:

| 範囲名           | ビットマスク   | 説明                                                                                    |
| ---------------- | ------------ | ---------------------------------------------------------------------------------------------- |
| ユニバーサルフラグ | `0xff000000` | すべてのトランザクションタイプに等しく適用されるフラグ。                                             |
| タイプベースのフラグ | `0x00ff0000` | 使用するトランザクションタイプによって意味が異なるフラグ。                |
| 予約済みフラグ   | `0x0000ffff` | 現時点では定義されていないフラグ。これらのフラグが無効である場合にのみ、トランザクションは有効です。 |

**注:** \[AccountSet transaction]\[]型には、type-based flagsと同様の目的を持つ独自の非ビット演算フラグがあります。レジャーオブジェクトにも、異なるビット演算フラグ定義を持つ `Flags` フィールドがあります。

### Memosフィールド

`Memos` フィールドには、トランザクションに付随する任意のメッセージデータが含まれます。これはオブジェクトの配列として表されます。各オブジェクトには `Memo` という1つのフィールドだけがあり、その中にさらに次のフィールドのうち1つ以上を含む別のオブジェクトがあります。

| フィールド   | 型     | \[内部型]\[] | 説明                                                                                                                                                                                                       |
| ------------ | ------ | ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MemoData`   | String | Blob                | 任意のhex値で、慣例的にはメモの内容を含みます。                                                                                                                                           |
| `MemoFormat` | String | Blob                | URLで許可される文字を表すhex値。慣例的には、メモのエンコード方法に関する情報を含みます。たとえば [MIMEタイプ](http://www.iana.org/assignments/media-types/media-types.xhtml)などです。 |
| `MemoType`   | String | Blob                | URLで許可される文字を表すhex値。慣例的には、[RFC 5988](http://tools.ietf.org/html/rfc5988#section-4)に基づく一意のrelationで、このメモの形式を定義します。                |

`MemoType` および `MemoFormat` フィールドは、次の文字のみで構成される必要があります: `ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-._~:/?#[]@!$&'()*+,;=%`

`Memos` フィールドのサイズは、1 KB以下に制限されています（バイナリ形式でシリアライズした場合）。

**トランザクションコスト:** `Memos` フィールドを持つトランザクションには、メモのサイズに基づく追加コストが発生します。コストは、トランザクション内のすべてのメモの合計サイズに比例して増加します。

`Memos` フィールドを持つトランザクションの例:

```json
{
    "TransactionType": "Payment",
    "Account": "rMmTCjGFRWPz8S2zAUUoNVSQHxtRQD4eCx",
    "Destination": "r3kmLJN5D28dHuH8vZNUZpMC43pEHpaocV",
    "Memos": [
        {
            "Memo": {
                "MemoType": "687474703a2f2f6578616d706c652e636f6d2f6d656d6f2f67656e65726963",
                "MemoData": "72656e74"
            }
        }
    ],
    "Amount": "1"
}
```

### NetworkIDフィールド

\[rippled 1.11.0で新規]\[]

`NetworkID` フィールドは "クロスチェーン" のトランザクション再生攻撃に対する保護であり、同じトランザクションがコピーされ、意図していない並列ネットワークで実行されるのを防ぎます。既存のチェーンとの互換性のため、`NetworkID` フィールドはNetwork IDが1024以下のネットワークでは省略しなければならず、Network IDが1025以上のネットワークでは含めなければなりません。次の表は、既知の各ネットワークの状態と値を示します:

| ネットワーク                     | ID    | `NetworkID` フィールド                                    |
| ------------------------------- | ----- | ---------------------------------------------------- |
| Mainnet                         | 0     | 不可                                           |
| Testnet                         | 1     | 不可                                           |
| Devnet                          | 2     | 不可                                           |
| AMM Devnet                      | 25    | 不可                                           |
| Sidechains Devnet Locking Chain | 2551  | 不可。ただし更新後に必須になります |
| Sidechains Devnet Issuing Chain | 2552  | 不可。ただし更新後に必須になります |
| Xahau Testnet                   | 21338 | 必須                                             |
| Xahau Mainnet                   | 21337 | 必須                                             |

トランザクションの再生攻撃は理論上可能ですが、2つ目のネットワーク上で特定の条件が必要です。次のすべてを満たす必要があります:

* トランザクションの送信者が、2つ目のネットワーク上で資金のあるアカウントであること。
* 2つ目のネットワーク上の送信者の `Sequence` 番号が、トランザクションの `Sequence` と一致しているか、またはトランザクションが2つ目のネットワークで利用可能なTicketを使用していること。
* トランザクションに `LastLedgerSequence` フィールドがないか、または2つ目のレジャーの現在のレジャーインデックスより大きい値を指定していること。
  * Mainnetは通常、テストネットワークやサイドチェーンよりもレジャーインデックスが大きいため、トランザクションが `LastLedgerSequence` を意図どおりに使用している場合、Mainnetのトランザクションをサイドチェーンやテストネットワークに再生するほうが、その逆より容易です。
* 2つのネットワークの両方のIDが1024以下であるか、両方のネットワークが同じIDを使用しているか、または2つ目のネットワークが `NetworkID` フィールドを必要としないこと。

### Signersフィールド

`Signers` フィールドにはマルチ署名が含まれ、最大8組の鍵ペアによる署名があり、それらが合わさってトランザクションを承認します。`Signers` リストはオブジェクトの配列で、各オブジェクトには `Signer` という1つのフィールドがあります。`Signer` フィールドには次のネストされたフィールドがあります:

| フィールド      | 型     | \[内部型]\[] | 説明                                                                   |
| --------------- | ------ | ------------------- | ----------------------------------------------------------------------------- |
| `Account`       | String | AccountID           | この署名に関連付けられたアドレスで、署名者リストに表示されるとおりのもの。 |
| `TxnSignature`  | String | Blob                | このトランザクションの署名で、`SigningPubKey` を使って検証できます。       |
| `SigningPubKey` | String | Blob                | この署名を作成するために使われた公開鍵。                                 |

`SigningPubKey` は、`Account` アドレスに関連付けられた鍵でなければなりません。参照先の `Account` がレジャー上の資金のあるアカウントであれば、`SigningPubKey` は、そのアカウントの現在のRegular Keyが設定されている場合、それを使用できます。また、`lsfDisableMaster` フラグが有効でない限り、そのアカウントのMaster Keyも使用できます。参照先の `Account` アドレスがレジャー上の資金のあるアカウントでない場合、`SigningPubKey` はそのアドレスに関連付けられたMaster Keyでなければなりません。

署名検証は計算負荷の高い処理であるため、マルチ署名トランザクションをネットワークへ中継するには追加のXAHが必要になります。マルチ署名に含まれる各署名は、そのトランザクションに必要な \[トランザクションコスト]\[]を増加させます。たとえば、トランザクションをネットワークへ中継するための現在の最低トランザクションコストが `10000` dropsであれば、`Signers` 配列に3エントリを持つマルチ署名トランザクションを中継するには、少なくとも `40000` dropsの `Fee` 値が必要になります。

### EmitDetailsフィールド

`EmitDetails` オブジェクトには次のフィールドがあります:

| フィールド        | JSON型   | \[内部型]\[] | 必須?     | 説明                                                                                                                                                                                                                                                                     |
| ----------------- | --------- | ------------------- | --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `EmitGeneration`  | Number    | UInt32              | はい      | このフィールドは、他のトランザクションの発行を引き起こす発行トランザクションの連鎖を追跡します。                                                                                                                                                                  |
| `EmitBurden`      | String    | UInt64              | はい      | このフィールドは、フォーク爆弾を検出するためのヒューリスティックです。手数料は負荷に基づいており、連鎖反応が始まったときに指数関数的に増加して、自己増強する発行トランザクションによってネットワークが圧倒されるのを防ぎます。                                                   |
| `EmitParentTxnID` | String    | Hash256             | はい      | トランザクションを発行したHook Executionは、発信トランザクションと接続されています。そのため、このフィールドは動作を効率的に追跡するために常に必要です。                                                                                                    |
| `EmitNonce`       | String    | Hash256             | はい      | 発行トランザクションは、同じフィールドを持つと同一になり、nonceを使用しなければ同じトランザクションハッシュを持ちます。ただし、ネットワーク上のすべてのノードがnonceについて合意する必要があるため、決定論的なnonceを生成するための特別なHook APIが用意されています。 |
| `EmitCallback`    | String    | AccountID           | いいえ    | このフィールドは、xahldがコールバックを開始する必要があるときに使用されます。これにより、どのHookとアカウントに対してコールバックを開始するかを把握できます。コールバックは、発行トランザクションがレジャーに受け入れられたときに発生します。                                                                 |
| `EmitHookHash`    | String    | Hash256             | はい      | 実行時点のHookのSHA512Hです。                                                                                                                                                                                                                            |

### Hookパラメーター

`HookParameters` フィールドは、Hookのパラメーターを指定するオブジェクトの配列です。各パラメーターオブジェクトには次のフィールドがあります。

| フィールド | JSON型 | 内部型 | 説明                 |
| ------- | --------- | ------------- | --------------------------- |
| `Name`  | String    | Blob          | パラメーターの名前。  |
| `Value` | String    | Blob          | パラメーターの値。 |

**トランザクションコスト:** `HookParameters` フィールドを持つトランザクションには、Hookパラメーターのサイズに基づく追加コストが発生します。コストは、トランザクション内のすべてのパラメーターの合計サイズに比例して増加します。
