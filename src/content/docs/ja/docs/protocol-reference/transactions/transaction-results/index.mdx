---
title: トランザクション結果
---
[\[ソース\]](https://github.com/ripple/rippled/blob/master/src/ripple/protocol/TER.h)

`rippled` サーバーは、`engine_result` や `meta.TransactionResult` のようなフィールドに現れる結果コードでトランザクション結果を要約します。これらのコードは、接頭辞ごとにいくつかのカテゴリに分かれています。

| カテゴリ              | 接頭辞 | 説明                                                                                                                                                                                          |
| --------------------- | ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| コストのみ消費        | `tec`  | トランザクションは目的を達成しませんでしたが、トランザクションコストは消費されました。この結果が最終確定となるのは、検証済みレジャーに含まれた場合のみです。 |
| 失敗                  | `tef`  | トランザクションは、サーバーの現在の（進行中の）レジャーにも、その後のどのレジャーにも適用できません。すでに適用済みであるか、あるいはレジャーの状態のため将来的にも適用不可能である可能性があります。 |
| ローカルエラー        | `tel`  | `rippled` サーバーが、高負荷などのローカル条件によりエラーを返しました。別のサーバーに再送信するか、別のタイミングで送信すると、異なる応答が返る場合があります。 |
| 不正なトランザクション | `tem`  | 不適切な構文、競合するオプション、誤った署名、その他の理由により、トランザクションが有効ではありませんでした。 |
| 再試行                | `ter`  | トランザクションは適用できませんでしたが、将来のレジャーでは正常に適用できる可能性があります。 |
| 成功                  | `tes`  | （エラーではありません）トランザクションは成功しました。この結果が最終確定となるのは、検証済みレジャーに含まれた場合のみです。 |

`rippled` サーバーは、失敗したトランザクションを自動的に再試行します。暫定的な失敗結果だけを見て、トランザクションが完全に失敗したと決めつけないことが重要です。成功または失敗が最終確定していない限り、後から成功する場合があります。

**警告:** トランザクションの暫定結果コードは、最終結果と異なる場合があります。暫定的に成功したトランザクションが最終的には失敗することもあれば、暫定的に失敗したトランザクションが最終的には成功することもあります。暫定的に失敗したトランザクションが、別のコードで最終的に失敗する場合もあります。トランザクション結果がいつ最終確定したと判断できるかについては、結果の確定性を参照してください。

ローカルエラー（`tel`）と不正なトランザクション（`tem`）の違いは、プロトコルレベルのルールにあります。たとえば、プロトコルには、トランザクションに含められるパス数の上限はありません。しかし、サーバー側では処理できるパス数に有限の制限を設けている場合があります。2つのサーバーが異なる設定になっていると、片方はパスの多いトランザクションに対して `tel` エラーを返し、もう片方は正常に処理できることがあります。十分な数のサーバーがそのトランザクションを処理でき、コンセンサスを通過すれば、検証済みレジャーに含めることができます。

これに対して、`tem` エラーは、設定に関係なく、どのサーバーでもそのトランザクションを適用できないことを意味します。つまり、トランザクションがプロトコルのルールに違反しているか、許容できないほど曖昧であるか、完全に意味をなしていません。不正なトランザクションが有効になる唯一の方法は、プロトコルそのものが変更されることです。たとえば、新しい機能が採用されると、その機能を使ったトランザクションは、まだその機能を知らない古いソフトウェアを実行しているサーバーでは不正と見なされることがあります。

### 即時レスポンス

`\[submitメソッド]\[]` の応答には、トランザクションのローカル処理中に何が起きたかを示す `rippled` サーバーの暫定結果が含まれます。

`submit` の応答には、次のフィールドが含まれます。

| フィールド                   | 値          | 説明                                                                                                                                              |
| ----------------------- | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `engine_result`         | String         | `tecPATH_DRY` など、トランザクションの結果を示すコードです。                                                                                          |
| `engine_result_code`    | Signed Integer | `engine_result` に対応する数値です。正確な値は予告なく変更される場合があります。                                                                       |
| `engine_result_message` | String         | 何が起きたかを説明する、人間が読めるメッセージです。このメッセージは開発者が問題を診断するためのもので、予告なく変更される場合があります。 |

トランザクションの送信とローカル適用で何も問題がなければ、応答は次のようになります。

```js
    "engine_result": "tesSUCCESS",
    "engine_result_code": 0,
    "engine_result_message": "The transaction was applied. Only final in a validated ledger."
```

**注:** この段階で成功結果が返っても、トランザクションが完全に成功したことを意味するわけではありません。ローカルサーバーが保持する暫定版レジャーに正常に適用されたことを示しているだけです。この段階での失敗結果も暫定的であり、変更される可能性があります。詳細は結果の確定性を参照してください。
