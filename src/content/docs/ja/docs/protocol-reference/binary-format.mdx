title: バイナリ形式
---
[\[出典\]](https://github.com/Xahau/xahaud/blob/dev/src/ripple/protocol/impl/STObject.cpp#L696-L718)

このページでは、Xahauのトランザクションおよびその他のデータに対する正規バイナリ形式を説明します。このバイナリ形式は、それらのトランザクション内容のデジタル署名を作成および検証するために必要であり、サーバー間のピアツーピア通信を含む他の場所でも使われます。`xahaud` APIは通常、クライアントアプリケーションとの通信にJSONを使用します。ただし、JSONはデジタル署名の対象となるトランザクションをシリアライズする形式としては不適切です。なぜなら、JSONは同じデータを複数の異なるが等価な方法で表現できるからです。

JSONやその他の表現からトランザクションを正規バイナリ形式へシリアライズする処理は、次の手順で要約できます。

1. 必須フィールドがすべて指定されていることを確認します。これには、必須だが「自動補完可能」なフィールドも含まれます。

    Transaction Formats Referenceには、Xahauのトランザクションに必要なフィールドと任意のフィールドが定義されています。

    **注意:** この段階では `SigningPubKey` も指定する必要があります。署名時には、署名用に与えられる秘密鍵からこの鍵を導出できます。
2. 各フィールドのデータを、その「内部」バイナリ形式に変換します。
3. フィールドを正規順序で並べ替えます。
4. 各フィールドの前にField IDを付けます。
5. フィールドを、前置情報も含めて並べ替え済みの順序で連結します。

その結果、ECDSA(secp256k1楕円曲線)やEd25519のような広く知られた署名アルゴリズムで署名できる単一のバイナリBlobが得られます。Xahauでは、適切なプレフィックス (`単一署名` の場合は `0x53545800`、`マルチ署名` の場合は `0x534D5400`)を付けてデータを \[hash]\[Hash]する必要もあります。署名後は、`TxnSignature` フィールドを含めてトランザクションを再シリアライズしなければなりません。

**注意:** Xahauは、レジャーオブジェクトや処理済みトランザクションなど、他の種類のデータを表現する際にも同じシリアライズ形式を使用します。ただし、署名されるトランザクションに含めるのに適したフィールドは一部だけです。（たとえば、署名そのものを含む `TxnSignature` フィールドは、署名対象のバイナリBlobに含めてはいけません。）そのため、いくつかのフィールドは「Signing」フィールドとして指定され、これらはオブジェクトが署名される際に含まれます。一方、「non-signing」フィールドは含まれません。

#### 例

署名済みトランザクションも未署名トランザクションも、JSON形式とバイナリ形式の両方で表現できます。以下のサンプルは、同じ署名済みトランザクションをJSON形式とバイナリ形式で示しています。

**JSON:**

```json

<div data-gb-custom-block data-tag="include" data-0='_code-samples/tx-serialization/py/test-cases/tx1.json'></div>

```

**バイナリ（16進表現）:**

```

<div data-gb-custom-block data-tag="include" data-0='_code-samples/tx-serialization/py/test-cases/tx1-binary.txt'></div>

```

### サンプルコード

ここで説明するシリアライズ処理は、複数の場所とプログラミング言語で実装されています。

* C++ の [`rippled` コードベース](https://github.com/Xahau/xahaud/blob/dev/src/ripple/protocol/impl/STObject.cpp)にあります。
* このリポジトリのコードサンプルセクションにあるJavaScript実装。
* このリポジトリのコードサンプルセクションにあるPython 3実装。

さらに、多くのクライアントライブラリが寛容なオープンソースライセンスの下でシリアライズ機能を提供しているため、コードをそのまま、または必要に応じて改変して利用できます。

### 内部形式

各フィールドには、署名時（およびほとんどの他の場面）にそのフィールドを表現するために `xahaud` ソースコードで使われる「内部」バイナリ形式があります。すべてのフィールドの内部形式は、[`SField.cpp`](https://github.com/Xahau/xahaud/blob/master/src/ripple/protocol/impl/SField.cpp)のソースコードで定義されています。（このファイルには、トランザクションフィールド以外のフィールドも含まれます。）Transaction Format Referenceには、すべてのトランザクションフィールドの内部形式も記載されています。

たとえば、共通トランザクションフィールドの `Flags` はUInt32（32ビットの符号なし整数）になります。

#### Field ID

[\[出典-エンコーディング\]](https://github.com/seelabs/rippled/blob/cecc0ad75849a1d50cc573188ad301ca65519a5b/src/ripple/protocol/impl/Serializer.cpp#L117-L148) [\[出典-デコーディング\]](https://github.com/seelabs/rippled/blob/cecc0ad75849a1d50cc573188ad301ca65519a5b/src/ripple/protocol/impl/Serializer.cpp#L484-L509)

フィールドの型コードとフィールドコードを組み合わせると、そのフィールドの一意な識別子が得られ、最終的なシリアライズ済みBlobではその前に付けられます。Field IDのサイズは、組み合わせる型コードとフィールドコードに応じて1バイトから3バイトです。次の表を参照してください。

|                      | 型コード < 16 | 型コード >= 16 |
| -------------------- | ------------- | -------------- |
| **フィールドコード < 16**  |               |                |
| **フィールドコード >= 16** |               |                |

デコード時には、**最初のバイト**のどのビットが0かによって、そのField IDが何バイトかを判別できます。これは上の表のケースに対応します。

|                            | 上位4ビットが非0                                                                                           | 上位4ビットが0                                                                 |
| -------------------------- | ----------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **下位4ビットが非0** | 1バイト:上位4ビットが型を定義し、下位4ビットがフィールドを定義します。                                                   | 2バイト:最初のバイトの下位4ビットがフィールドを定義し、次のバイトが型を定義します        |
| **下位4ビットが0**    | 2バイト:最初のバイトの上位4ビットが型を定義し、最初のバイトの下位4ビットは0、次のバイトがフィールドを定義します | 3バイト:最初のバイトは `0x00`、2番目のバイトが型を定義し、3番目のバイトがフィールドを定義します |

**注意:** Field IDはフィールドを並べ替えるために使う2つの要素で構成されていますが、Field IDのバイト構造によって並び順が変わるため、シリアライズ済みのField IDそのものでソートしてはいけません。

#### 長さプレフィックス

可変長フィールドの一部の型には、長さを示す値が前置されます。`Blob` フィールド（任意のバイナリデータを含む）はその一例です。どの型に長さプレフィックスが付くかは、Type Listの表を参照してください。

**注意:** 長さが変わるフィールド型の中には、長さプレフィックスを持たないものもあります。それらの型では、内容の終端を示す別の方法が使われます。

長さプレフィックスは、型プレフィックスの直後、内容の前にあるフィールドの長さを示す1バイトから3バイトで構成されます。

* フィールドが0から192バイトのデータを含む場合、最初のバイトが内容の長さを示し、その長さ分のデータが長さバイトの直後に続きます。
*   フィールドが193から12480バイトのデータを含む場合、最初の2バイトが次の式で長さを示します。

    ```
      193 + ((byte1 - 193) * 256) + byte2
    ```
*   フィールドが12481から918744バイトのデータを含む場合、最初の3バイトが次の式で長さを示します。

    ```
      12481 + ((byte1 - 241) * 65536) + (byte2 * 256) + byte3
    ```
* 長さプレフィックス付きフィールドは、918744バイトを超えるデータを含めることはできません。

デコード時には、最初の長さバイトの値から、追加の長さバイトが0個、1個、または2個あるかを判別できます。

* 最初の長さバイトの値が192以下なら、それが唯一の長さバイトであり、フィールド内容の正確なバイト長を表します。
* 最初の長さバイトの値が193から240なら、長さバイトは2個です。
* 最初の長さバイトの値が241から254なら、長さバイトは3個です。

### 正規フィールド順

トランザクション内のすべてのフィールドは、まずフィールドの型（各型に割り当てられた数値の「型コード」）で、その後にフィールド自体（「フィールドコード」）で決まる特定の順序で並べ替えられます。（これは、姓で並べ、次に名で並べるようなものだと考えてください。ここで姓がフィールドの型、名がフィールド自体です。）

#### 型コード

各フィールド型には任意の型コードがあり、数値が小さいものが先に並びます。これらのコードは [`SField.h`](https://github.com/Xahau/xahaud/blob/master/src/ripple/protocol/SField.h#L57-L74)で定義されています。

たとえば、[UInt32の型コードは2](https://github.com/Xahau/xahaud/blob/72e6005f562a8f0818bc94803d222ac9345e1e40/src/ripple/protocol/SField.h#L59)なので、すべてのUInt32フィールドは [Amountフィールド（型コード6）](https://github.com/Xahau/xahaud/blob/72e6005f562a8f0818bc94803d222ac9345e1e40/src/ripple/protocol/SField.h#L63)より前に並びます。

定義ファイルには、`TYPES` マップ内で各型の型コードが一覧されています。

#### フィールドコード

各フィールドにはフィールドコードがあり、同じ型を持つフィールド同士を並べるために使われます。数値が小さいものが先に並びます。これらのフィールドは [`SField.cpp`](https://github.com/Xahau/xahaud/blob/72e6005f562a8f0818bc94803d222ac9345e1e40/src/ripple/protocol/impl/SField.cpp#L72-L266)で定義されています。

たとえば、\[Payment transaction]\[]の `Account` フィールドには [ソートコード1](https://github.com/Xahau/xahaud/blob/72e6005f562a8f0818bc94803d222ac9345e1e40/src/ripple/protocol/impl/SField.cpp#L219)があるため、[ソートコード3](https://github.com/Xahau/xahaud/blob/72e6005f562a8f0818bc94803d222ac9345e1e40/src/ripple/protocol/impl/SField.cpp#L221)を持つ `Destination` フィールドより前に来ます。

フィールドコードは異なるフィールド型間で再利用されますが、同じ型のフィールドが同じフィールドコードを持つことはありません。型コードとフィールドコードを組み合わせると、そのフィールドの一意なField IDになります。

#### AccountIDフィールド

この型のフィールドは、XRP Ledgerアカウントの160ビット識別子を含みます。JSONでは、これらのフィールドは追加のチェックサムデータ付きの \[base58]\[] XRP Ledger "アドレス" として表されるため、誤記が有効なアドレスになる可能性は低くなります。（このエンコード方式は、しばしば "Base58Check" と呼ばれ、誤って別のアドレスに送金してしまうことを防ぎます。）これらのフィールドのバイナリ形式にはチェックサムデータが含まれず、アドレスのbase58エンコードで使われる `0x00` の "type prefix" も含まれません。（ただし、バイナリ形式は主に署名済みトランザクションに使われるため、署名済みトランザクションを書き写す際の誤記やその他の誤りは署名を無効にし、送金を防ぎます。）

単独フィールドとして現れるAccountID（たとえば `Account` や `Destination`）は、160ビット固定長であるにもかかわらず長さプレフィックスが付きます。そのため、これらのフィールドの長さ指示子は常にバイト `0x14` です。特殊フィールドの子要素として現れるAccountID（Amountの `issuer` とPathSetの `account`）は、長さプレフィックスを持ちません。

#### Amountフィールド

`Amount` 型は、XAHまたはトークンのいずれかの通貨金額を表す特殊なフィールド型です。この型は2つのサブタイプから成ります。

*   XAH

    XAHは64ビットの符号なし整数（ビッグエンディアン順）としてシリアライズされます。ただし、最上位ビットはXAHであることを示すため常に0、上から2番目のビットは正の値であることを示すため `1` です。XAHの最大量（1017 drops）は57ビットしか必要としないため、標準の64ビット符号なし整数に `0x4000000000000000` をビットごとのORで適用することでXAHのシリアライズ形式を計算できます。
*   **トークン**

    トークンは、次の3つのセグメントをこの順で含みます。

    1. トークン金額形式での金額を示す64ビット。最初のビットは、これがXAHではないことを示すために `1` です。
    2. 通貨コードを示す160ビット。標準APIは "USD" のような3文字コードを標準通貨コード形式で160ビットコードに変換しますが、独自の160ビットコードも可能です。
    3. 発行者のAccount IDを示す160ビット。（関連項目: Account Address Encoding）

最初のビットに基づいて、どちらのサブタイプかを判別できます。`0` ならXAH、`1` ならトークンです。

**トークン金額形式**

[\[出典\]](https://github.com/Xahau/xahaud/blob/35fa20a110e3d43ffc1e9e664fc9017b6f2747ae/src/ripple/protocol/impl/STAmount.cpp)

include_svg("img/currency-number-format.svg", "トークン金額形式の図")

Xahauは、(fungible)トークンの数値金額をシリアライズするために64ビットを使います。（JSON形式では、数値金額は通貨金額オブジェクトの `value` フィールドです。）バイナリ形式では、その数値金額はこの順で、"not XAH" ビット、符号ビット、有効数字、指数から構成されます。

1. トークン金額の最初の（最上位）ビットは、それがXAH金額ではないことを示すために `1` です。（XAH金額は常に最上位ビットが `0` に設定され、この形式と区別されます。）
2. 符号ビットは、金額が正か負かを示します。標準の [two's complement](https://en.wikipedia.org/wiki/Two's_complement)整数とは異なり、Xahau形式では `1` が **正**、`0` が負を示します。
3. 次の8ビットは、指数を符号なし整数として表します。指数はスケール（有効数字に掛ける10のべき乗）を示し、範囲は-96から +80（両端含む）です。ただし、シリアライズ時には指数に97を加えて、符号なし整数として表せるようにします。そのため、シリアライズ値 `1` は指数 `-96`、シリアライズ値 `177` は指数80を示し、以下同様です。
4. 残りの54ビットは、有効数字（ _mantissa_ とも呼ばれます）を符号なし整数として表します。シリアライズ時には、この値は0の特別ケースを除き、1015 (`1000000000000000`)から1016-1 (`9999999999999999`)の範囲に正規化されます。0の特別ケースでは、符号ビット、指数、有効数字はすべて0となり、64ビット値は `0x8000000000000000000000000000000000000000` としてシリアライズされます。

数値金額は、通貨コードおよび発行者とともにシリアライズされ、完全なトークン金額を形成します。

**通貨コード**

プロトコルレベルでは、Xahauの通貨コードは任意の160ビット値ですが、次の値には特別な意味があります。

* 通貨コード `0x0000000000000000000000005852500000000000` は **常に禁止** です。（これは "標準形式" における "XAH" のコードです。）
* 通貨コード `0x0000000000000000000000000000000000000000`（すべて0）は **通常は禁止** です。通常、XAH金額は通貨コードで指定されません。ただし、このコードは、あるフィールドがXAHの通貨コードを指定しなければならないまれな場合に、XAHを示すために使われます。

`xahaud` APIは、3文字のASCIIコードを160ビットの16進値へ変換する **標準形式** を次のようにサポートしています。

include_svg("img/currency-code-format.svg", "標準通貨コード形式")

1. 最初の8ビットは `0x00` でなければなりません。
2. 次の88ビットは予約済みで、すべて `0` である必要があります。
3. 次の24ビットはASCIIの3文字を表します。Rippleは、[ISO 4217](https://www.xe.com/iso4217.php)コード、または "BTC" のような一般的なpseudo-ISO 4217コードの使用を推奨しています。ただし、次の文字の任意の組み合わせが許可されます。大文字および小文字のアルファベット、数字、ならびに `?`、`!`、`@`、`#`、`$`、`%`、`^`、`&`、`*`、`<`、`>`、`(`、`)`、`{`、`}`、`[`、`]`、`|` の各記号です。通貨コードXAH（すべて大文字）はXAH用に予約されており、トークンには使用できません。
4. 次の40ビットは予約済みで、すべて `0` である必要があります。

**非標準形式** は、先頭8ビットが `0x00` でない限り、任意の160ビットデータです。

#### 配列フィールド

`SignerEntries`（\[SignerListSet transactions]\[]内）や `Memos` のような一部のトランザクションフィールドは、オブジェクトの配列（"STArray" 型）です。

配列には、ネイティブのバイナリ形式における複数のオブジェクトフィールドが、特定の順序で含まれます。JSONでは、配列の各メンバーは単一のフィールドを持つJSONのラッパーオブジェクトであり、そのフィールド名がメンバーのオブジェクトフィールド名です。そのフィールドの値は、そのオブジェクト自身（内部オブジェクト）です。

バイナリ形式では、配列の各メンバーはField IDプレフィックス（ラッパーオブジェクトの単一キーに基づく）と内容（内部オブジェクトをオブジェクトとしてシリアライズしたもの）を持ちます。配列の終わりを示すには、`0xf1`（フィールドコード1の配列に対する型コード）という "Field ID" を持ち、内容を持たない項目を追加します。

#### Blobフィールド

Blob型は、任意のデータを含む長さプレフィックス付きフィールドです。この型を使う代表的なフィールドは `SigningPubKey` と `TxnSignature` で、それぞれトランザクションの実行を承認する公開鍵と署名を含みます。

Blobフィールドの内容にはそれ以上の構造がないため、Field IDと長さプレフィックスの後に、可変長エンコードで示されたバイト数ちょうどから成ります。

#### Hashフィールド

Xahauには、Hash128、Hash160、Hash256といういくつかの "hash" 型があります。これらのフィールドは、指定されたビット数の任意のバイナリデータを含み、ハッシュ演算の結果を表す場合もしない場合もあります。

これらのフィールドはすべて、長さ指示子なしで、指定されたビット数そのままをビッグエンディアンのバイト順でシリアライズします。

#### Issueフィールド

一部のフィールドは、金額を持たないアセットの _type_ 、つまりXAHまたは代替可能トークンを指定します。これらのフィールドは、順に1個または2個の160ビットセグメントで構成されます。

1. 最初の160ビットは、そのアセットの通貨コードです。XAHの場合、これはすべて0です。
2. 最初の160ビットがすべて0（アセットがXAH）なら、そこでフィールドは終わりです。そうでない場合、そのアセットはトークンであり、次の160ビットはトークン発行者のAccountIDです。

#### オブジェクトフィールド

`SignerEntry`（\[SignerListSet transactions]\[]内）や、`Memos` 配列内の `Memo` のような一部のフィールドは、オブジェクト（"STObject" 型）です。オブジェクトのシリアライズは配列と非常によく似ていますが、1つ違いがあります。**オブジェクトメンバーは、オブジェクトフィールド内で正規順序に配置する必要がある** ことです。一方、配列フィールドにはすでに明示的な順序があります。

オブジェクトフィールドの正規フィールド順は、すべてのトップレベルフィールドに対する正規フィールド順と同じですが、オブジェクトのメンバーはオブジェクト内で並べ替える必要があります。最後のメンバーの後には、内容を持たない `0xe1` の "Object end" Field IDが続きます。

次の例は、オブジェクト（`Memos` 配列内の1つの `Memo` オブジェクト）のシリアライズ形式を示しています。

#### PathSetフィールド

クロスカレンシー \[Payment transaction]\[]の `Paths` フィールドは "PathSet" であり、JSONでは配列の配列として表されます。パスの用途については、Pathsを参照してください。

PathSetは、**1から6** 個の個別パスを順にシリアライズしたものです[\[出典\]](https://github.com/Xahau/xahaud/blob/4cff94f7a4a05302bdf1a248515379da99c5bcd4/src/ripple/app/tx/impl/Payment.h#L35-L36)。完全な各パスの後には、次に何が来るかを示すバイトが続きます。

* `0xff` は次のパスが続くことを示します
* `0x00` はPathSetの終わりを示します

各パスは、順に **1から8** 個のパスステップで構成されます[\[出典\]](https://github.com/Xahau/xahaud/blob/4cff94f7a4a05302bdf1a248515379da99c5bcd4/src/ripple/app/tx/impl/Payment.h#L38-L39)。各ステップは **type** バイトで始まり、その後にパスステップを表す1つ以上のフィールドが続きます。typeは、ビット単位のフラグを通じて、そのパスステップにどのフィールドが含まれるかを示します。（たとえば、値 `0x30` はcurrencyとissuerの両方が変更されることを示します。）複数のフィールドがある場合、それらは常に特定の順序で配置されます。

次の表は、可能なフィールドと、それらを示すためにtypeバイトに設定するビットフラグを説明しています。

| 型フラグ | 含まれるフィールド | フィールド型   | ビット長  | 順序 |
| -------- | ------------------ | -------------- | -------- | ---- |
| `0x01`   | `account`          | AccountID      | 160 bits | 1番目 |
| `0x10`   | `currency`         | Currency Code  | 160 bits | 2番目 |
| `0x20`   | `issuer`           | AccountID      | 160 bits | 3番目 |

いくつかの組み合わせは無効です。詳細はPath Specificationsを参照してください。

`account` と `issuer` フィールドのAccountIDは、長さプレフィックスなしで表されます。`currency` がXRPの場合、通貨コードは160ビットのすべて0として表されます。

各ステップは、その次のステップに直接続きます。上で説明したように、パスの最後のステップの後には、`0xff`（別のパスが続く場合）または `0x00`（これが最後のパスの終わりの場合）のいずれかが続きます。

次の例は、PathSetのシリアライズ形式を示しています。

#### UIntフィールド

Xahauには、UInt8、UInt16、UInt32、UInt64といういくつかの符号なし整数型があります。これらはすべて、指定されたビット数を持つ標準的なビッグエンディアンのバイナリ符号なし整数です。

JSONオブジェクトでこれらのフィールドを表すとき、ほとんどは既定でJSON数値として表されます。例外の1つはUInt64で、これは一部のJSONデコーダーが64ビットの倍精度浮動小数点数として表そうとする可能性があるため、文字列で表されます。これらの浮動小数点数では、すべての異なるUInt64値を完全な精度で表現できません。

もう1つの特別なケースは `TransactionType` フィールドです。JSONでは、このフィールドは通常、トランザクションタイプ名の文字列として表されますが、バイナリではUInt16です。定義ファイルの `TRANSACTION_TYPES` オブジェクトは、これらの文字列を特定の数値に対応付けます。
