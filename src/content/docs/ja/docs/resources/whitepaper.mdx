## 1. 要約

1. Xahau Ledger（Xahau）は、XRPL エコシステム向けのスマートコントラクト・サイドチェーンです。これは XRP Ledger（XRPL）のオープンソース `rippled` コードベースをフォークしたもので、環境持続性を含む XRPL の有用で革新的な特性をすべて引き継ぎつつ、スマートコントラクトをサポートするようコードベースを調整・拡張しています。
2. Xahau の中核機能は次の通りです:
    * **XRPL Core:** Xahau は、**XRP Ledger Consensus Protocol**（旧称: Ripple Protocol Consensus Algorithm）、**DEX**、およびネイティブトークンで手数料を徴収し焼却することでスパムや肥大化からレジャーを保護するロジックなど、XRPL を長く人気あるネットワークにしてきた重要機能を保持しています。一方で、比較的新しい **XLS-20 NFTs** は、よりクリーンでシンプルな **URITokens** に置き換えています。
    * **Hooks for Smart Contracts:** Xahau の大きな新機能は、`rippled` 向けスマートコントラクト実装である Hooks です。Hooks はアカウントにインストールされる小さなコード片で、そのアカウントが送受信するトランザクションに対し、レジャーで確定する前にルールを適用します。
    * **Native Token & Better Tokenomics:** Xahau はネイティブトークン XAH（通貨コード: XAH）によって保護され、バリデーターへの報酬とスマートコントラクト支援を意図した、より良いトークノミクスによって駆動されます。
    * **Genesis Hook Governance Game:** Xahau の Genesis アカウントは Hook によって制御されており、とりわけ新規 XAH 発行を管理しています。この Hook は、最大 20 の独立所有バリデーターが参加する 2 層構造の governance game によって統治されます。
3. これらの機能が組み合わさることで、XRPL エコシステム内に新しいネットワークが生まれます。Hooks amendment を実装することで、適切にインセンティブ付けされたバリデーターと開発者のコミュニティに支えられた、高速・低コスト・安全なスマートコントラクト環境を XRPL エコシステムへ提供します。
4. Xahau Ledger の開発は、XRPL コミュニティ内の経験豊富で献身的な 5 つの主体からなる Xahau Launch Alliance によって進められてきました。これらの主体は ICO を行うことなく、新しいネットワークの立ち上げに必要なコストとリスクを担ってきました。
5. Xahau は、開発者、取引所、長期コミュニティメンバーを含む XRPL エコシステムを代表する構成の 10 を超える独立運営バリデーターにより、完全に機能し分散化された状態でローンチされます。

## 2. Xahau – XRPL の DNA から構築

1. Xahaud、すなわち Xahau を動かすソフトウェアは、XRPL のオープンソースコードベース `rippled` のフォークです。2012 年のローンチ以来、XRPL を最も人気があり長寿なネットワークの一つにしてきた主要機能を維持しています。

### XRP Ledger Consensus Protocol

2. Xahau は XRP Ledger Consensus Protocol（旧称: Ripple Protocol Consensus Algorithm）を採用しています。これは Proof of Association（PoA）とも呼ばれることがあります。PoW や PoS のような従来のコンセンサスメカニズム、特に PoW に伴う高いエネルギー消費の問題を克服するよう設計されています。
3. XRP Ledger Consensus Protocol（旧称: Ripple Protocol Consensus Algorithm）は次のように機能します:
    * **Validator Proposal:** 各バリデーターは、次のレジャーに含めるべきだと考える新しいトランザクション集合、いわゆる「candidate set」を提案します。この candidate set には新規トランザクションと、まだ検証済みレジャーに含まれていないトランザクションの両方が含まれます。
    * **Agreement Phase:** バリデーター同士が通信し、candidate set を交換・評価します。他のバリデーターが提案したトランザクションの妥当性と順序を独立して検証します。反復的なラウンドを経て、多数派が支持する単一の candidate set へ収束しようとします。
    * **Finalization:** バリデーターの超多数（少なくとも 80%）が特定の candidate set に同意すると、その集合は「finalized」と見なされます。finalized された candidate set が次のレジャーの基礎になります。
    * **Ledger Closing:** レジャーがクローズされ、finalized された candidate set 内のトランザクションに基づいて新しいレジャーが作られます。XRPL ではこの処理が概ね 3〜5 秒ごとに発生し、高速なトランザクション確定を可能にしています。
4. XRP Ledger Consensus Protocol（旧称: Ripple Protocol Consensus Algorithm）の下では、各バリデーターは自分が信頼する他バリデーター一覧である UNL（Unique Node List）を指定します。正規のレジャーは、各バリデーターの UNL が重なり合う部分を通じて見つけられます。
5. 適切な UNL に関する社会的合意を支援するため、XRPL Foundation を含む複数の主体が、十分に信頼できると見なしたバリデーター一覧、すなわち推奨 Validator Lists（VLs）を公開しています。一般に受け入れられている dUNL に従うことで、メインレジャーに留まり、無関係なフォークを誤って追従してしまうことを避けやすくなります。Xahau も同様に、XRPL Foundation のような組織が推奨 VL を公開する社会的合意ベースの dUNL 定義アプローチを取ります。
6. XRP Ledger Consensus Protocol（旧称: Ripple Protocol Consensus Algorithm）は、高い拡張性、低エネルギー消費、検閲やネットワーク分岐への耐性といった利点を提供します。信頼されたバリデーター集合とこの独自コンセンサスアルゴリズムにより、Xahau は分散性と信頼性を保ちながら高速かつ安全なトランザクション処理を実現します。

### DEX

7. Xahau は、利用者がレジャー上で直接資産売買・交換できる XRPL の指値注文型分散型取引所（DEX）の一形態を保持しています。この DEX は、Xahau の分散型かつトラストレスな設計を活かし、仲介者なしのピアツーピア資産取引を可能にします。
8. 要約すると、この分散型取引所は次のように動作します:
    * **Order Creation:** 利用者は、売買したい資産、数量、受け入れるまたは支払う価格を指定して DEX 上に買い注文または売り注文を作成できます。
    * **Order Matching:** DEX のオーダーブックは、未約定の買い注文と売り注文をすべて記録します。新しい注文が入ると、価格と数量に基づいて既存注文と自動的にマッチングされます。
    * **Cross-Currency Trading:** DEX は異なる資産間のクロスカレンシー取引をサポートします。そのために、レジャーのネイティブ通貨をブリッジ通貨として用い、ある資産からネイティブ通貨を経由して別資産へ変換できるようにします。
    * **Pathfinding Function:** pathfinding は DEX の重要な構成要素です。ある資産から別の資産へ変換する際、最も効率的な取引経路を見つけるのに役立ちます。たとえば Asset A を Asset D へ交換したいが直接市場がない場合、A から B、B から C、C から D のような最適な連続取引を探索し、スリッページと手数料を最小化します。
    * **Automatic Order Execution:** 取引がマッチすると、DEX は自動的にトランザクションを実行します。レジャーのコンセンサスメカニズムにより、そのトランザクションは検証されレジャーへ追加され、高い安全性と不変性が確保されます。
    * **Trustless and Decentralized:** DEX はトラストレスかつ分散型で動作します。中央集権型取引所に依存せず、ユーザー資産を第三者ウォレットへ預ける必要もありません。利用者は取引全体を通して資産の管理権を保持し、カウンターパーティリスクを減らせます。
    * **Low Fees and Fast Settlement:** DEX 上の取引手数料は、従来の中央集権型取引所に比べて最小限です。Xahau のコンセンサスプロトコルにより高速な決済が可能で、ほぼ即時のトランザクション処理を実現します。
9. これらの特徴により、DEX はピアツーピアのトークン取引に便利かつ安全な基盤を提供します。Xahau のトラストレスかつ分散型という性質を活かし、利用者のコントロールとプライバシーを維持したまま、効率的なクロストークン取引とシームレスな注文マッチングを実現します。

### 焼却されるトランザクション手数料

10. スパムやレジャー肥大化から自らを守るため、XRPL はネイティブ通貨でトランザクション手数料とアカウントリザーブを徴収し、その手数料をバリデーターへ再分配するのではなく全額焼却します。Xahau も同様にこの「徴収して焼却する」モデルを採用して保護を行います。
11. これはレジャー保護の実績ある仕組みですが、後述するように Xahau のトークノミクスに影響を与えます。Xahau のスマートコントラクトで焼却される手数料は依然として小さいとはいえ、XRPL 上の通常手数料よりは大きく、しかも変動幅があります。

### XLS-20 NFTs の代わりに URIToken NFTs

12. XRPL に導入された XLS-20 NFT 標準が NFT をレジャー上の「ページ」へ圧縮して保持するのに対し、Xahau は URIToken オブジェクトを使います。
13. URIToken（Uniform Resource Identifier Token の略）は、NFT、そのメタデータ、所有情報を Xahau 上で表現・管理するための革新的な方法です。各トークンは、現在の所有者が変わっても変化しない一意のアドレスを持つ第一級のレジャーオブジェクトです。
14. URIToken NFTs は XLS-20 NFT 標準に対して多くの利点があります:
    * **Lightweight and Efficient:** URITokens は第一級のレジャーオブジェクトであり、作成、破棄、移転、売買が可能で、Hooks とも容易に相互運用できます。取引はトークンごとに単一の売りオファーに限定され、そのオファーは指定された送達先アカウント、または送達先未指定なら一般公開先によって受け入れられます。仲介モードはありません。
    * **Interoperability and Immutability:** URITokens には拡張可能なメタデータ JSON 標準と、NFT 内容を不変化する任意の Digest フィールド（ハッシュ）が備わっています。これにより、開発者やクリエイターは IPFS や従来型 Web サーバーなど好みの保存方法を選べます。また、既存 NFT 標準やプラットフォームとの相互運用を促進し、より広い NFT エコシステムとの統合を容易にします。
    * **Improved User Experience:** URITokens は、現在の所有者に依存しない keylet（オンレジャーアドレス）で簡単に特定できます。NFT のメタデータと所有情報は、標準 Web プロトコルで容易に取得・更新できます。利用者は NFT に関連付けられた URI をクリックするだけで、画像、説明、来歴などの詳細を閲覧できます。この方式は、Xahau 上で NFT を扱うユーザー体験を簡素化し、複雑さを減らします。
15. このように、URITokens は Xahau 上の NFT に対して軽量で柔軟かつコスト効率のよい解決策を提供します。

## 3. Hooks – XRPL エコシステム向けスマートコントラクト

1. Hooks は、XRPL エコシステム向けに特化して開発されたスマートコントラクトソリューションです。Hooks は Xahau アカウント上に定義される小さく効率的なコード片で、そのアカウントへ送信またはそのアカウントが受信するトランザクションに対し、レジャー上で確定する前にロジックを実行します。そのため Hooks は、開発者が Xahau 上にスマートコントラクトを作成・展開するための手段であり、分散型アプリケーション（dApps）や自動化トランザクションに幅広い可能性をもたらします。
2. Xahau は Hooks amendment をサイドチェーンとして実装することで、XRPL エコシステムが Hooks の恩恵を今すぐ受けられるようにしています。同時に、将来 XRPL メインネットに Hooks を実装するべきか、その方法をどうするかをコミュニティがよりよく評価できるようにもなります。

### Hooks の利点

3. Hooks が他のスマートコントラクトソリューションより優れている点として、以下が挙げられます:
    * **Simplicity and Efficiency:** Hooks は従来型スマートコントラクトソリューションと比べて軽量で効率的です。トランザクション決済と高速コンセンサスに重点を置く Xahau の基盤設計により、性能を損なうことなく高速にスマートコントラクトを実行できます。
    * **Native Integration:** Hooks はレイヤー 1 のスマートコントラクトソリューションであり、レジャー内部にネイティブに存在します。そのため、開発者はオンレジャーオブジェクト、残高、トランザクションと直接やり取りするコントラクトを作れます。レイヤー 2 型のスマートコントラクトソリューションと異なり、この密接な統合により開発が簡単になり、Xahau エコシステムとのシームレスな相互運用性が確保されます。
    * **Low Fees and Scalability:** Hooks は Xahau の低手数料と高い拡張性を活用します。多くのチェーンと同様、Xahau は毎秒大量のトランザクションを処理でき、高速かつ低コストなスマートコントラクト実行を必要とするアプリケーションに適しています。
    * **Security and Reliability:** Hooks は XRPL の堅牢なセキュリティと信頼性を受け継ぎます。Xahau を支える XRP Ledger Consensus Protocol（旧称: Ripple Protocol Consensus Algorithm）は、信頼されたバリデーター間の合意に基づく分散型ネットワークを実現します。この強固な基盤は、開発者と利用者の双方に安心感を与えます。
    * **Ecosystem Compatibility:** Hooks は既存の XRPL ツールやサービスと互換性を持つよう設計されており、開発者は自身のアプリケーションへスマートコントラクトを組み込みやすくなります。この互換性により、分散型取引所やペイメントチャネルなど XRPL の他機能ともシームレスに連携できます。
    * **Account Decentralisation:** Hooks を使うと、Xahau アカウントは完全自律的に動作できるようになります。外部の秘密鍵保有者が手動承認する代わりに、Hook のロジックに従ってトランザクションを送出・受信できます。

### 想定ユースケース

4. XRPL エコシステムへスマートコントラクト機能を導入することで、Hooks は分散型アプリケーション構築やトランザクション自動化を目指す開発者に強力かつ効率的なソリューションを提供します。Hooks によって実現できる機能の例としては次のようなものがあります:
    * **Whitelists:** Hook を有効化したアカウントは、不正または制裁対象のトランザクションからアカウントを保護するため、ホワイトリスト登録済みアカウントとの送受信のみを受け入れるようコード化できます。
    * **Blacklists:** Hook を有効化したアカウントは、禁止された（ブラックリスト化された）アカウントとの送受信を拒否するようコード化できます。
    * **Sophisticated Escrow:** Hook を有効化したアカウントは、指定されたアカウントから受け取った任意資産を保管し、事前定義条件を満たす有効トランザクションによって指示された場合にのみ、その資産を転送または返却できます。
    * **Automated Registry:** Hook を有効化したアカウントは自動登録簿として機能し、登録または登録権の証拠として URI NFT を mint / redeem できます。
    * **Self-Sovereign Treasury:** 発行通貨は、自身が所有するアカウントへ預け入れられ、そのアカウントに設定された Hook が公開済みかつ事前定義された発行スケジュールに従って通貨を emit することで、分散型でカウンターパーティ不要の資産のように振る舞えます。
5. シンプルさ、低コスト、拡張性、安全性、エコシステム互換性の組み合わせにより、Xahau は XRPL エコシステムへ魅力的なスマートコントラクトソリューションへの早期アクセスを提供します。Hooks を用いた Xahau の経験は、将来 XRPL で Hooks を実装すべきか、またどう実装すべきかをコミュニティが判断する助けになります。

## 4. XAH – ネイティブトークンとトークノミクス

1. Xahau はネイティブトークン XAH によって保護されています。主な仕様は次の通りです:
    * **Native Token:** XAH
    * **Ticker:** XAH
    * **Smallest Unit:** 1 drop または 0.000001 XAH
    * **Function:** XAH はネットワークの通貨です。ネットワークサービスを購入するためのユーティリティトークンです。XRPL と同様に、Xahau 上のトランザクションではレジャーをスパムや肥大化から守るために XAH 建ての手数料またはリザーブが課されます。
    * **Initial Liquidity:** ローンチ参加者へ 6 億が分配されます。
    * **Max Supply:** 上限なし。
    * **Emission Mechanisms:** XAH の供給量は、Xahau のコンセンサスプロトコルで有効化された amendment を通じて決定されます:
    * **Monthly Balance Adjustment:** トランザクション手数料として焼却された XAH の一部を再流通させるため、各アクティブユーザーは自身の月間平均残高の約 0.34%（年率 4% 複利相当）の残高増加を請求できます。この発行は、Hooks、BalanceRewards、XahauGenesis amendments を組み合わせて利用する Protocol Governance Game によって制御されます。
    * **Monthly Seat Rewards:** トランザクション手数料として焼却された XAH の一部を再流通させると同時に、積極的かつ適切なガバナンスを促進するため、請求された Monthly Balance Adjustments の合計額と同額が Governance Game バリデーターへ報酬として与えられます。

### 初期配布

2. ローンチ当事者は Xahau の立ち上げに多大な時間と資源を投じてきました。初期配布は XahauGenesis amendment によりプロトコルレベルで提供されます。
3. プロトコルは、控えめな 6 億 XAH を以下のように支払います: ローンチ時に 8 つの Governance Game validator Seats それぞれへ 1200 万。
    * DEX のステーブルコイン流動性立ち上げのため GateHub へ追加で 1600 万。
    * 知的財産対価として XRPL Labs（Xaman）へ追加で 1 億 6000 万。Xaman は Hooks と xahaud 開発の中心的な技術的・財務的負担を担ってきました。この追加配布はその貢献に対するものです。
    * XRPL Foundation（現 INFTF）へ 3 億 2800 万。今後の XRPL Protocol（現 Xahau）エコシステムの健全性を支えるためです。

### Monthly Balance Adjustment

4. Xahau が管理しなければならない課題の 1 つは、通常の XRPL トランザクションと比べたスマートコントラクトの追加コストです。
5. XRPL では、トランザクション手数料は 1 drop、つまり 0.000001 XRP という非常に低い額です。総供給量 1000 億 XRP に対して見れば、これはごくわずかですが十分なコストです。一方 Hooks トランザクションは、設定にも発火にも桁違いに高いコストがかかり得ます。さらに、XRPL の各トランザクションが標準化されているのに対し、Hooks は複雑性もコストも大きく異なります。Hooks 由来のスパムや肥大化から適切に守るには、各 Hook トランザクションのサイズとロジック複雑性に応じて適切かつ動的に課金する必要があります。
6. その結果、手数料を焼却するスマートコントラクトチェーンとして、XAH は極端な場合ほぼゼロまで焼却される可能性があります。少なくとも、XRPL 上の XRP よりはるかに速いペースで減少し得ます。そうなると、Hooks の展開や実行に使うより XAH をため込む方が有利になるという望ましくないインセンティブが生まれるかもしれません。
7. こうしたリスクを避けるため、Xahau には焼却済みトークンを再流通させるプロトコルレベルの仕組みが必要です。ただし、単純に高手数料を奨励するものではいけません。これが monthly balance adjustment の目的です。
8. Monthly Balance Adjustment は次のように機能します:
    * 毎月、各 Xahau アカウントは、自身の平均 XAH 残高（前回調整以降の平均）に基づく調整額をプロトコルに対して請求できます。調整額は年率 4% 複利相当（およそ月 0.34%）です。
    * 利用者は、前回請求から 30 日経過していればいつでも Genesis アカウント宛に RewardClaim トランザクションを送ることで balance adjustment を請求できます。これにより、その利用者がアクティブであることが示されます。
    * 請求トランザクションを受け取ると、プロトコルは BalanceRewards および XahauGenesis amendments を通じて、その利用者アカウント残高を調整額だけ増加させます。
    * 請求されなかった balance adjustment は、非アクティブに対するペナルティとして失効します。
    * 4% はローンチ時の初期率ですが、将来小さすぎるまたは大きすぎると判断された場合には Governance Game によって変更できます。
9. この仕組みにより、アクティブユーザーは月次調整を請求して追加の XAH を流通供給へ戻せるため、スマートコントラクト実行手数料によってトークンがゼロへ近づくリスクを和らげられます。

### Monthly Governance Validator Rewards

10. Xahau Genesis Account には Hook が設定されており、とりわけ XAH の発行を制御しています。この Hook は、変化するネットワーク状況や現実世界の条件に応じて監視、修正、または置換できるようなガバナンス体制を必要とします。
11. XRPL メインネットのバリデーターは、メインネットレジャーを支えるサービスに対して報酬を受け取っていないことがよく知られています。しかし、インフラ運用には費用がかかり、現実にはこのインセンティブ不足が、積極的に管理されるバリデーター不足につながります。
12. Xahau Genesis Hook は、自身を更新するか判断する際にどのアカウントを信頼すべきかを知っている、あるいは知らされる必要があります。そのため、それらのアカウントは UNL の重なりから自然発生するのではなく、あらかじめ分かっている必要があります。
13. この問題に対する技術的解決策が、特定のバリデーターへ Hook 監督の報酬を与えるインセンティブ付き Genesis Hook Governance Game です。これは次のように動作します:
    * Genesis Hook Governance Game には 20 の「Seats」があります。
    * 各 Seat は、単一の Xahau Account（Level 1）または 3〜20 アカウントから成る自己管理型委員会（Level 2）のいずれかによって制御されます。
    * 各占有 Seat に対し、プロトコルはその月に請求された Monthly Balance Adjustments 合計の 1/20 を mint して分配します。
    * Seat がアカウント委員会によって保有されている場合、その Seat の報酬分配方法はその委員会が決定します。
    * その月に誰も Balance Adjustment を請求しなければ、どのバリデーターにも報酬は支払われません。
    * 各 Flag Ledger（256 レジャー）ごとに、UNLReport と呼ばれるオンレジャー記録が生成されます。このレポートには、前回 Flag Ledger 以降に 80% の他バリデーターが十分にコンセンサスへ参加したと考えるバリデーターの公開鍵一覧が含まれます。Genesis Hook は、UNLReport に掲載されたバリデーター公開鍵に対応する Seats に対してのみ報酬を配布します。
14. したがって、Genesis Hook Governance Game には、信頼に値するバリデーターを運用する信頼済みアカウントへ報酬を与え、その報酬をユーザー活動と結びつけ、新たな XAH を流通へ戻す仕組みが組み込まれています。これにより、高いスマートコントラクト実行手数料によるトークン流動性逼迫リスクを軽減します。

## 5. Xahau Governance Game
1. XRP Ledger Consensus Protocol（旧称: Ripple Protocol Consensus Algorithm）を採用するチェーンは、正規レジャーを定義するために、バリデーター UNL の強い重なりを必要とします。あるバリデーターが、他の人々が信頼しているのと十分に同じマシンを信頼していない場合、無関係なレジャーフォークを誤って追従してしまう危険があります。最適な UNL を決めることは、社会的（オフチェーン）合意の問題です。
2. その社会的合意を助けるため、さまざまな主体が dUNL、すなわち皆が従うべきバリデーター一覧を公開しています。Xahau も同じ方針を取り、XRPL Foundation などがそのような一覧を公開することを想定しています。したがって、Xahau も XRPL と同様に、誰でもバリデーターを運用できるパブリックかつパーミッションレスなチェーンになります。また、XRPL と同様に、プロトコル更新は UNL バリデーターの 80% 超による支持を必要とします。
3. dUNL は正規レジャーに責任を持つバリデーターを定義するのに役立ちますが、Xahau には XRPL メインネットにはない第 2 のガバナンス課題があります。それが Genesis Hook のガバナンスです。XRPL メインネットは作成時にすべての XRP を mint 済みでしたが、Xahau は同じ方式を採れません。ネイティブトークンは、Burn2Mint、Account Adjustments、Validator Rewards といった定義済みイベントに応じて mint される必要があり、そのために Genesis Account 上の Hook を使用します。
4. これは、Hook の調整や置換が必要になった場合に備えて、Xahau に Genesis Hook 向けガバナンスメカニズムが必要であることを意味します。これが Governance Game です:
    * **Purpose:** Governance Game の目的は、UNL バリデーターがアクティブな状態を保ち、それらアクティブメンバーが XAH 分配を制御する Genesis account 上の Hook を保守できるようにすることです。
    * **Seats:** テーブルには 20 の Seats があります。Seat とは、Governance Game において投票権を持つ Xahau Account のことです。
    * **L1 Table:** Governance Game には少なくとも上位テーブルが必要です。これを Level 1 table（L1 Table）と呼びます。このテーブルは定義上、Genesis account 上にインストールされた Governance Game Hook にのみ存在します。L1 table の seat は Account によって埋められることも、空席であることもあります。これらの seat は L1s と呼ばれます。
    * **L2 Table:** L1 seat は、さらに別の Governance Hook がインストールされた別アカウントによって埋められることがあります。このテーブルの seats は L2s、そのテーブルは L2 Table と呼ばれます。これは 3〜20 アカウントからなる自己管理型委員会で、L1 seat とその投票権を共同で制御します。したがって、L1 と L2 の混在次第で、最大 400 アカウントが 2 層ガバナンスゲームに関与できます。
    * **Filling Seats:** L1 Seats は、埋まっている全 L1 Seats の 80% 投票によって就任または解任されます。L2 Seats については、その L1 Seat を共同保有するアカウント委員会が、多数決または自ら適切と考える別ロジックにより、新メンバーを招待または除名します（最大 20、最小 3）。他の Seats は L2 Table のメンバー構成に一切関与しません。
    * **Voting:** 各 L1 Seat は Governance Game で 1 票を持ちます。L2 Seats では、その L2 table の単一票をどのように決めるかはその L2 table 自身が決定します。デフォルトは、その L2 table の L2 Seat メンバー過半数です。
    * **Hook Changes:** Hook へのすべての変更は Governance Game を通じて投票されます。Hook 変更は、定義された割合の Seats が支持した場合にのみ成立します。通常投票率は 80% ですが、balance adjustment 率の引き上げについては 100% が必要です。
    * **Rewards:** L1 Seats が信頼性の高いバリデーターを運用するインセンティブとして、コンセンサスへ積極参加している限り、L1 Seats は月次 Rewards を得ます。
    * **Foundation of dUNL**: Genesis Hook Governance Game に参加する者たちが運用するバリデーターは、任意の dUNL の基盤を成すべきです。信頼性の低い Seat は他 Seats の投票で追い出されるリスクがあるためです。
5. 異なる種類の Seats を設けることで、安全性を損なわずにより多様なアカウント参加を可能にしています。Level 1 Seats はコミュニティが信頼できる大きな主体に適しており、Layer 2 Seats はより広い参加層を許容します。ただし L2 の集合参加は 1 票として扱われるため、Level 1 Seats に置かれる信頼を圧倒することはありません。
6. したがって、Governance Game は最大 400 アカウントの参加を可能にし、それらアカウントが信頼性の高いバリデーターを運用してコンセンサスにも参加している限り、報酬を与えます。

## 6. Xahau Launch Alliance

1. Xahau は、XRPL エコシステム上で構築と支援の実績を持つ 5 つの独立主体から成る連合によって開発されました。構成主体は次の通りです:
    * XUMM ウォレット、Hooks、Xahaud の開発元であり、Xahau 開発の専門性とコストの大部分を担ってきた XRPL Labs（Xaman）
    * XRPL 上のステーブルコイン発行も行う、多国籍テクノロジー企業・暗号資産取引所・暗号資産サービス提供者である GateHub Limited
    * 安全なホスティング提供に特化し、XRPL インフラの主要提供者（Alloy Networks として）でもある IT コンサルティングおよびインフラ企業 Titanium OU
    * Evernode スマートコントラクトプロジェクトの開発元であり、その Xahau への展開を担った Evernode Labs Ltd
    * 会計、監査、法務サービスを世界規模で提供する企業に関連する法人、Digital Governing OU

## 7. 完全機能かつ分散化された状態でローンチ

1. Xahau は資金調達を行わず、完全機能かつ分散化された状態でローンチされます。中央集権的支配はありません。
2. Xahau は 20 の Governance Game Seats のうち 8 を埋めた状態でローンチされます:
    * 5 つは Layer 1 Seats で、それぞれ Xahau Launch Alliance のメンバーが保持します。
    * 3 つは Layer 2 Tables で、そのうち 1 つは取引所委員会、別の 1 つは XRP コミュニティのプロジェクト、開発者、長期支援者から成る多様な委員会が保持します。
    * 1 つは、Hooks を含むブロックチェーン開発向けコード監査・レビューサービスを提供する FYEO が保持します。

### 資金調達は行われない

3. ICO はなく、ローンチに伴う資金調達も行われません。
4. Xahau Alliance の各メンバーは、それぞれ独立して自らの開発努力へ資金を投じてきました。共同資金も、Xahau の追加開発に向けた資金・資源の持ち寄りもありません。また、ロードマップや「Xahau Foundation」、その他の中央集権的な財源も存在しません。今後の開発は、エコシステム内の独立主体の裁量に完全に委ねられます。

### 約束はない

5. Governance Game のメンバーは独立主体です。どのように投票するか、どのように資源を配分するか、XAH を含む資産をどう使うかについて、相互の合意、取り決め、または了解はありません。「他メンバーから見て信頼できるバリデーターであり続けなければ、ゲームから投票排除される」という Game の論理だけが、バリデーター行動を駆動します。これは Xahau が用いる XRP Ledger Consensus Protocol（旧称: Ripple Protocol Consensus Algorithm）（PoA）の本質的な特徴です。

### 追加機能は不要

6. ソフトウェアはその性質上保守が必要ですが、Xahaud を XRPL のスマートコントラクト・サイドチェーンとして有用にするうえで欠けている本質的機能や、必須の追加開発項目は存在しません。
