## 1. 概要

1. Xahau Ledger（Xahau）は、XRPLエコシステム向けのスマートコントラクト用サイドチェーンです。XRP Ledger（XRPL）のオープンソース`rippled`コードベースを基にしており、XRPLの環境持続性を含む有用で革新的な機能を受け継ぎつつ、スマートコントラクトを支えるためにコードベースを調整・拡張しています。
2. Xahauの中核機能は次のとおりです。
    * **XRPLコア:** Xahauは、XRPLを最も長く利用され、人気の高いネットワークのひとつにしてきた主要機能を保持しています。これには **XRP Ledger Consensus Protocol**（以前: Ripple Protocol Consensus Algorithm）、**DEX**、およびネイティブトークンで手数料を課しバーンすることでレジャーをスパムや肥大化から守る仕組みが含まれます。一方で、最近の **XLS-20 NFT** は、より明快でシンプルな **URIToken** に置き換えています。
    * **スマートコントラクト用Hooks:** Xahauの大きな新機能は、`rippled`向けのスマートコントラクト実装であるHooksです。Hooksはアカウントに導入される小さなコード片で、アカウントが送信または受信するトランザクションに対して、確定前にルールを適用します。
    * **ネイティブトークンと改良されたトークノミクス:** XahauはネイティブトークンXAH（通貨コード: XAH）で保護され、バリデーターへの報酬とスマートコントラクトの支援を目的とした改良されたトークノミクスで動作します。
    * **Genesis Hookガバナンスゲーム:** XahauのGenesisアカウントは、XAHの新規発行などを制御するHookによって動作しており、このHookは、最大20の独立した所有バリデーターが参加する2層構造のガバナンスゲームによって運営されます。
3. これらの機能を組み合わせることで、XRPLエコシステム内に新しいネットワークが生まれ、Hooks amendmentを実装して、適切なインセンティブを受けたバリデーターと開発者コミュニティに支えられた、高速・低コスト・安全なスマートコントラクトを提供します。
4. Xahau Ledgerの開発は、Xahau Launch Allianceによって推進されてきました。彼らは、ICOなしで新しいネットワークを立ち上げるコストとリスクを負担した、XRPLコミュニティ内の経験豊富で献身的な5つの組織です。
5. Xahauは、開発者、取引所、長期コミュニティメンバーを含むXRPLエコシステムの代表的な構成によって運用される、10を超える独立したバリデーターとともに、完全に機能する分散型ネットワークとしてローンチします。

## 2. Xahau - XRPLのDNAから構築

1. Xahauを動かすソフトウェアであるxahaudは、XRPLのオープンソースコードベース`rippled`のフォークです。2012年に公開されて以来、XRPLを最も人気があり長寿なネットワークのひとつにした主要機能を保持しています。

### XRP Ledger Consensus Protocol

2. Xahauは、XRP Ledger Consensus Protocol（以前: Ripple Protocol Consensus Algorithm）を採用しています。これはProof of Association（PoA）とも呼ばれ、Proof of Work（PoW）やProof of Stake（PoS）のような従来型コンセンサスの制約、特にPoWに伴う高いエネルギー消費を克服するために設計されています。
3. XRP Ledger Consensus Protocol（以前: Ripple Protocol Consensus Algorithm）は、次のように動作します。
    * **バリデーターの提案:** 各バリデーターは、次のレジャーに含めるべきだと考える新しいトランザクション集合を、「candidate set」と呼ばれる形で提案します。このcandidate setには、新しいトランザクションと、以前はvalidated ledgerに含まれていなかったトランザクションの両方が含まれます。
    * **合意フェーズ:** バリデーター同士がcandidate setを交換・評価し、他のバリデーターが提案したトランザクションの妥当性と順序を個別に検証します。反復的なラウンドを通じて、過半数が合意する1つのcandidate setへ収束することを目指します。
    * **確定:** バリデーターの超多数（少なくとも80%）が特定のcandidate setに同意すると、その集合は「finalized」と見なされます。finalizedされたcandidate setが次のレジャーの基盤になります。
    * **レジャークローズ:** レジャーがクローズされ、finalizedされたcandidate setのトランザクションに基づいて新しいレジャーが作成されます。この処理はXRPLでは概ね3〜5秒ごとに行われ、高速なトランザクション決済を可能にします。
4. XRP Ledger Consensus Protocol（以前: Ripple Protocol Consensus Algorithm）では、各バリデーターがUNL（Unique Node List）を指定します。これは、そのバリデーターが信頼する他のバリデーターの一覧です。canonical ledgerは、すべてのバリデーターのUNLの重なりから導かれます。
5. 適切なUNLに関する社会的合意を支えるため、XRPL Foundationを含む複数の組織が、自ら信頼できると判断したバリデーターの推奨Validator List（VL）を公開しています。一般に受け入れられているdUNLに従うことで、メインレジャーにとどまり、意図せず無関係なフォークを追う事態を避けやすくなります。Xahauも同様に、XRPL Foundationのような組織が推奨VLを公開する社会的合意ベースのdUNL定義を採用します。
6. XRP Ledger Consensus Protocol（以前: Ripple Protocol Consensus Algorithm）は、高いスケーラビリティ、低エネルギー消費、検閲やネットワークフォークへの耐性など、多くの利点を備えています。信頼されたバリデーターの集合とこの独自のコンセンサスアルゴリズムにより、Xahauは分散性と信頼性を維持しながら、高速かつ安全なトランザクション処理を実現します。

### DEX

7. Xahauは、XRPLのlimit order book型のdecentralized exchange（DEX）の一部を継承しており、ユーザーがレジャー上で直接アセットを取引・交換できるようにします。DEXは、Xahauの分散型かつtrustlessなアーキテクチャを活用し、仲介者なしのピアツーピア資産取引を可能にします。
8. decentralized exchangeの動作は次のとおりです。
    * **注文作成:** ユーザーは、取引したいアセット、数量、受け入れたいまたは支払いたい価格を指定して、DEX上で売り注文または買い注文を作成できます。
    * **注文マッチング:** DEXのorder bookは、すべての未約定の売買注文を記録します。新しい注文が出されると、DEXは価格と数量に基づいて既存注文と自動的に照合します。
    * **クロスカレンシー取引:** DEXは異なるアセット間の取引をサポートしており、これによりユーザーは複数の資産をまたいで取引できます。そのために、DEXはレジャーのネイティブ通貨をブリッジ通貨として使い、ある資産をネイティブ通貨に、さらに別の資産へと変換できるようにします。
    * **パス探索機能:** パス探索機能はDEXの重要な構成要素です。あるアセットを別のアセットに変換する際、最も効率的な取引経路を見つけるのに役立ちます。たとえば、Asset AをAsset Dに交換したいが直接の市場がない場合、パス探索機能は最適な取引の連鎖（例: A→B、B→C、C→D）を探し、スリッページと手数料を最小限に抑えて変換を実現します。
    * **自動注文執行:** 取引が一致すると、DEXはトランザクションを自動的に実行します。レジャーのコンセンサスメカニズムにより、そのトランザクションは検証されレジャーに追加され、高い安全性と不変性が確保されます。
    * **trustlessかつ分散型:** DEXはtrustlessかつ分散型の仕組みで動作します。中央集権的な取引所に依存せず、ユーザーが資金を第三者ウォレットへ預ける必要もありません。その代わり、取引の全過程で資産の管理権はユーザーが維持するため、カウンターパーティリスクが低減します。
    * **低手数料と高速決済:** DEXでの取引は、従来の中央集権型取引所と比べて手数料が非常に低く抑えられます。Xahauのコンセンサスプロトコルにより決済が高速化され、ほぼ即時のトランザクション処理が可能になります。
9. これらの機能により、DEXはユーザーにピアツーピアのトークン取引のための便利で安全なプラットフォームを提供します。Xahauのtrustlessかつ分散型という特性を活かして、DEXはユーザーの管理権とプライバシーを維持しながら、効率的なトークン間取引とシームレスな注文マッチングを実現します。

### バーンされるトランザクション手数料

10. スパムや肥大化から自衛するため、XRPLはネイティブ通貨でトランザクション手数料とアカウントリザーブを課し、バリデーターへ再分配するのではなく、すべてのトランザクション手数料をバーンします。Xahauも同様に、この「手数料を課し、バーンする」モデルを採用して自らを保護します。
11. これはレジャー保護の実績ある仕組みですが、後述するようにXahauのトークノミクスには影響があります。Xahauのスマートコントラクトでバーンされる手数料は、依然として小額ではあるものの、XRPLの標準トランザクション手数料よりはるかに高く、変動幅も大きくなります。

### XLS-20 NFTsではなくURIToken NFTs

12. XRPLに導入されたXLS-20 NFT標準は、NFTをレジャー上の「pages」に圧縮して保持しますが、XahauはURITokenオブジェクトを使用します。
13. URITokens（Uniform Resource Identifier Tokenの略）は、Xahau内でNFT、そのメタデータ、所有情報を表現・管理するための革新的な方法です。各トークンは第一級のレジャーオブジェクトであり、現在の所有者が変わっても変化しない固有のアドレスを持ちます。
14. URIToken NFTはXLS-20 NFT標準に比べて多くの利点があります。
    * **軽量で効率的:** URITokensは第一級のレジャーオブジェクトであり、作成、破棄、移転、購入、販売が可能です。またHooksとも容易に連携します。取引は各トークンにつき1つのsell offerに限定されており、そのofferは指定されたdestinationアカウント、またはdestinationが指定されていない場合は一般公開で受け入れられます。brokered modeはありません。
    * **相互運用性と不変性:** URITokensには拡張可能なメタデータJSON標準が組み込まれており、任意のDigestフィールドもあります。DigestはNFTの内容を不変にするハッシュです。これにより、開発者やクリエイターはIPFSや従来のWebサーバーなど、好みの保存方法を選べます。また、既存のNFT標準やプラットフォームとの相互運用性も高まり、より広いNFTエコシステムへの統合が容易になります。
    * **改善されたユーザー体験:** URITokensは、トークンの現在の所有者に依存しないkeylet（レジャー上のアドレス）により簡単に見つけられます。NFTのメタデータと所有情報は、標準的なWebプロトコルで簡単に取得・更新できます。ユーザーはNFTに関連付けられたURIをクリックするだけで、画像、説明、来歴などの詳細を確認できます。この方法は、ユーザー体験を簡素化し、Xahau上でNFTを扱う複雑さを軽減します。
15. したがって、URITokensはXahau上のNFTに対する軽量で柔軟、かつ費用効率の高い解決策を提供します。

## 3. Hooks - XRPLエコシステム向けスマートコントラクト

1. Hooksは、XRPLエコシステム向けに特化して開発されたスマートコントラクトソリューションです。Hooksは、Xahauアカウント上に定義された小さく効率的なコードで、レジャーに確定される前に、そのアカウントが送受信するトランザクションに対してロジックを実行します。これにより開発者はXahau上でスマートコントラクトを作成・展開でき、分散型アプリケーション（dApps）や自動化トランザクションの幅広い可能性が開かれます。
2. XahauはHooks amendmentをサイドチェーンとして実装しているため、XRPLエコシステムは将来ではなく今すぐHooksの恩恵を受けられます。また、将来的にXRPLメインネットへHooksをどのように、そして実装すべきかをコミュニティがよりよく評価できるようになります。

### Hooksの利点

3. Hooksが他のスマートコントラクトソリューションより優れている主な点は次のとおりです。
    * **シンプルさと効率性:** Hooksは、従来のスマートコントラクトソリューションに比べて軽量かつ効率的です。トランザクション決済と高速コンセンサスに重点を置くXahauの基盤アーキテクチャにより、性能を損なわずにスマートコントラクトを迅速に実行できます。
    * **ネイティブ統合:** Hooksはレイヤー1のスマートコントラクトソリューションであり、レジャー内部にネイティブに存在します。これにより開発者は、レジャー上のオブジェクト、残高、トランザクションと直接やり取りするコントラクトを作成できます。レイヤー2ソリューションと比べて、この緊密な統合は開発を簡素化し、Xahauエコシステムとのシームレスな相互運用を実現します。
    * **低手数料とスケーラビリティ:** HooksはXahauの低いトランザクション手数料と高いスケーラビリティを活用します。多くのチェーンと同様に、Xahauは高い秒間トランザクション数を処理できるため、高速かつコスト効率の高いスマートコントラクト実行を必要とするアプリケーションに適しています。
    * **セキュリティと信頼性:** HooksはXRPLの堅牢なセキュリティと信頼性を継承します。Xahauを支えるXRP Ledger Consensus Protocol（以前: Ripple Protocol Consensus Algorithm）は、信頼されたバリデーター間のコンセンサスによる分散型ネットワークを保証します。この強固な基盤は、開発者とユーザーの双方に安心感を与えます。
    * **エコシステム互換性:** Hooksは既存のXRPLツールやサービスと互換性を持つよう設計されており、開発者がアプリケーションにスマートコントラクトを統合しやすくなっています。この互換性により、分散型取引所やpayment channelなど、XRPLの他機能ともシームレスに連携できます。
    * **アカウントの分散化:** Hooksを使うことで、Xahauアカウントは完全に自律的になり、外部の秘密鍵保有者による手動承認に頼るのではなく、Hookのロジックに従ってトランザクションを発行・受信できます。

### 代表的なユースケース

4. XRPLエコシステムにスマートコントラクト機能を導入することで、Hooksは分散型アプリケーションの構築やトランザクション自動化を目指す開発者に強力で効率的な解決策を提供します。Hooksが実現できる機能の例は次のとおりです。
    * **ホワイトリスト:** Hook対応アカウントは、ホワイトリストに登録されたアカウントとのみ送受信するようにコード化することで、不正や制裁対象トランザクションからアカウントを保護できます。
    * **ブラックリスト:** Hook対応アカウントは、禁止された（ブラックリスト化された）アカウントとの間でトランザクションを送受信しないようにコード化できます。
    * **高度なEscrow:** Hook対応アカウントは、指定されたアカウントから受け取ったあらゆるアセットを保持し、事前条件を満たす有効なトランザクションにより指示された場合にのみ、その資産を送信または返却できます。
    * **自動レジストリ:** Hook対応アカウントは、自動レジストリとして動作し、登録または登録権の証拠としてURI NFTをmint / redeemできます。
    * **自己主権型トレジャリー:** 発行済み通貨は、開示済みかつ事前に定められた発行スケジュールに従って通貨を発行するHookを備えた自己所有アカウントに預けることで、分散型かつカウンターパーティ不要のアセットを模倣できます。
5. シンプルさ、低手数料、スケーラビリティ、セキュリティ、エコシステム互換性の組み合わせにより、XahauはXRPLエコシステム向けの魅力的なスマートコントラクトソリューションへの早期アクセスを提供します。Hooksに関するXahauの実運用経験は、コミュニティが将来XRPLにHooksを実装するかどうか、またどのように実装するかを判断する助けになります。

## 4. XAH -ネイティブトークンとトークノミクス

1. XahauはネイティブトークンXAHによって保護されています。主な内容は次のとおりです。
    * **ネイティブトークン:** XAH
    * **ティッカー:** XAH
    * **最小単位:** 1 dropまたは0.000001 XAH
    * **機能:** XAHはネットワークの通貨です。ネットワークサービスを購入するためのユーティリティトークンです。XRPLと同様に、Xahau上のトランザクションには、レジャーをスパムや肥大化から守るため、XAH建ての手数料またはリザーブが課されます。
    * **初期流動性:** ローンチ参加者に6億枚を配布。
    * **最大供給量:** 上限なし。
    * **発行メカニズ�:** XAHの供給量は、Xahauのコンセンサスプロトコルで有効化されたamendmentによって決定されます。
    * **月次残高調整:** トランザクション手数料でバーンされたXAHの一部を再び流通に戻すため、各アクティブユーザーは自分の平均月次残高の約0.34%の増加を請求できます（年率4%の複利相当）。この発行は、Hooks、BalanceRewards、XahauGenesisの各amendmentを組み合わせて動作するProtocol Governance Gameによって制御されます。
    * **月次シート報酬:** トランザクション手数料でバーンされたXAHの一部を再び流通に戻し、アクティブで適切なガバナンスを促すため、その月に請求されたMonthly Balance Adjustmentの合計と同額がGovernance Gameのバリデーターにも報酬として配分されます。

### 初期配布

2. ローンチ参加者は、Xahauを立ち上げるために多大な時間とリソースを投じてきました。初期配布はXahauGenesis amendmentによってプロトコルレベルで提供されます。
3. プロトコルは、合計6億Xahau XAHを次のように配布します。ローンチ時にGovernance Gameの8つのバリデーター席それぞれへ1,200万XAH。
    * GateHubには、DEXにおけるステーブルコイン流動性の初期構築のために追加で1,600万XAH。
    * XRPL Labs（Xaman）には、知的財産に対して追加で1億6,000万XAH。XamanはHooksとxahaudの開発において、主たる技術的・財務的負担を担ってきました。この追加配布はその貢献に対するものです。
    * XRPL Foundation（現INFTF）には3億2,800万XAH。これは、XRPL Protocol（現Xahau）エコシステムの健全性を今後も維持するための支援です。

### 月次残高調整

4. Xahauが対処しなければならない課題の1つは、通常のXRPLトランザクションと比べてスマートコントラクトの方が追加コストがかかることです。
5. XRPLでは、トランザクション手数料は1 drop、つまり0.000001 XRPからです。1000億XRPという総供給量に照らせば、これは小さいが十分なコストです。一方でHooksのトランザクションは、設定や発火に桁違いのコストがかかる場合があります。さらに、XRPLの各トランザクションが標準化されているのに対し、Hooksは複雑さとコストが大きく異なります。Hooksによるスパムや肥大化から適切に保護するには、各Hookトランザクションのサイズとロジックの複雑さに応じて、正しく動的に課金する必要があります。
6. その結果、手数料をバーンするスマートコントラクトチェーンとして、XAHは容易にほぼゼロ近くまでバーンされ得ます。これはXRPLにおける同等のXRPよりもはるかに速く進むため、Hooksをデプロイ・実行する代わりにXahauをため込むという逆インセンティブを生みうるのです。
7. これらのリスクを防ぐため、Xahauには、バーンされたトークンを流通に戻す一方で、単に高い手数料を促進しないプロトコルレベルの仕組みが必要です。それが月次残高調整の目的です。
8. 月次残高調整は次のように機能します。
    * 毎月、各Xahauアカウントは、自身の平均XAH残高（前回の調整以降に算出）に基づく調整を、プロトコルに対して請求できます。調整額は、年率4%の複利（毎月およそ0.34%）に相当します。
    * ユーザーは、前回の請求から30日が経過していれば、いつでもRewardClaimトランザクションをGenesisアカウントに送ることで残高調整を請求できます。これにより、そのユーザーがアクティブであることを示せます。
    * 請求トランザクションを受け取ると、プロトコルはBalanceRewardsとXahauGenesisの各amendmentを通じて、ユーザーのアカウント残高を調整額分だけ増加させます。
    * 請求されなかった残高調整は、非アクティブの罰則として失効します。
    * 4%はローンチ時の初期レートであり、将来的に小さすぎるまたは大きすぎると判断された場合はGovernance Gameにより変更できます。
9. この仕組みにより、アクティブなユーザーは毎月の調整を請求して流通供給量にさらにXAHを加えられ、スマートコントラクト実行手数料によってトークンがゼロまでバーンされるリスクを軽減できます。

### 月次ガバナンスバリデーター報酬

10. Xahau Genesis Accountには、XAHの発行などを制御するHookが有効化されています。このHookは、ネットワークや現実世界の条件が変化したときに、監視・修正・置換できるよう、ガバナンスの仕組みを必要とします。
11. XRPLメインネットのバリデーターは、メインネットレジャーを支えるために提供するサービスに対して、基本的に何らかの報酬を受け取ることはないことが広く知られています。しかし、インフラの運用には費用がかかり、現実世界の経験は、インセンティブが欠けると積極的に運用されるバリデーターが減ることを示しています。
12. Xahau Genesis Hookは、自身を更新するかどうかを判断する際に、どのアカウントを信頼すべきかを知る必要があります。したがって、それらのアカウントは、重なり合うUNLから自然発生するのではなく、事前に把握されていなければなりません。
13. この問題に対する技術的解決策は、Hookの監督に対して特定のバリデーターへ報酬を与える、インセンティブ付きのGenesis Hook Governance Gameです。仕組みは次のとおりです。
    * Genesis Hook Governance Gameには20の「席」があります。
    * 各席は、単一のXahauアカウント（Level 1）または3〜20のアカウントからなる自己管理委員会（Level 2）によって管理されます。
    * 埋まっている各席について、プロトコルはその月に請求されたMonthly Balance Adjustmentの合計の1/20をmintして配布します。
    * 席がアカウント委員会によって保持されている場合、その席の報酬はその委員会の選択した方法で分配されます。
    * その月に残高調整を請求するユーザーがいなければ、どのバリデーターにも報酬は支払われません。
    * 各Flag Ledger（256レジャー）ごとに、UNLReportと呼ばれるレジャー上の記録が生成されます。このレポートには、前回のFlag Ledger以降にコンセンサスへ十分参加したと他のバリデーターの80%が判断したバリデーターの公開鍵一覧が含まれます。Genesis Hookは、UNLReportに載っているバリデーターの公開鍵に対応する席へだけ報酬を分配します。
14. したがって、Genesis Hook Governance Gameには、信頼されたアカウントが信頼できるバリデーターの運用に対して報われること、報酬がユーザー活動に連動すること、そしてその報酬によって新しいXAHが流通に加わり、高額なスマートコントラクト実行手数料によるトークン流動性不足のリスクを抑えることを目的とした報酬メカニズムが含まれています。

## 5. Xahauガバナンスゲーム
1. XRP Ledger Consensus Protocol（以前: Ripple Protocol Consensus Algorithm）系のチェーンは、canonical ledgerを定義するために、バリデーターのUNL間の十分な重なりに依存しています。あるバリデーターが、他の皆が信頼しているのと同じマシンを十分に信頼していない場合、無関係なレジャーフォークを気づかずに追ってしまうリスクがあります。最適なUNLを決めることは、オフチェーンの社会的合意の問題です。
2. その社会的合意を支えるため、さまざまな組織がdUNLを公開しています。dUNLは、誰もが従うべきバリデーター一覧です。Xahauも同じアプローチを取り、XRPL Foundationなどがこのような一覧を公開すると想定されています。したがってXRPLと同様に、Xahauも誰でもバリデーターを立てられる公開・許可不要のチェーンになります。またXRPLと同様に、プロトコル更新は、XRPLと同じやり方で、UNLバリデーターの80%超が変更を支持することに依存します。
3. dUNLはcanonical ledgerを支えるバリデーターを定義する助けになりますが、XahauにはXRPLメインネットにはない第2のガバナンス課題があります。それはGenesis Hookのガバナンスです。XRPLメインネットは、XRPを作成時にすべてmintしましたが、Xahauは同じ方法を取れません。ネイティブトークンは、Burn2Mint、Account Adjustments、Validator Rewardsなど、あらかじめ定義された事象に応じてmintされる必要があります。その実現にGenesis Account上のHookを使っています。
4. つまり、Hookを調整または置換する必要がある場合に備えて、XahauにはGenesis Hookのガバナンス機構が必要です。これがGovernance Gameです。
    * **目的:** Governance Gameは、UNLバリデーターがアクティブな状態を保つことを保証し、アクティブなメンバーがXAHの配布を制御するGenesis account上のHookを維持できるようにするために存在します。
    * **席:** テーブルには20席あります。Seatは、Governance Gameにおける投票が有効なXahau Accountです。
    * **L1テーブル:** Governance Gameには少なくとも最上位テーブルが必要です。これをLevel 1 table（L1 Table）と呼びます。このテーブルは、そこにインストールされたGovernance Game Hookに従い、定義上Genesis accountの上にのみ存在します。L1 tableの席は、アカウントで埋めても空席でも構いません。これらの席はL1と呼ばれます。
    * **L2テーブル:** L1の席は、それ自体が別の異なるGovernance Hookを備えたアカウントで埋めることができます。このテーブルの席はL2と呼ばれ、テーブル自体はL2 Tableです。この構造は、L1の席とその投票権を共同で管理する3〜20アカウントの自己管理委員会です。したがって、L1とL2の席の組み合わせに応じて、最大400アカウントが2層のGovernance gameに関わる可能性があります。
    * **席の充足:** L1の席は、現在埋まっているL1席の80%の票により、埋めることも空けることもできます。L2の席では、そのL1の席を共同で保持するアカウント委員会が、過半数投票、または委員会が妥当とみなすその他のロジックやルールに従って、新メンバーの招待・非招待を行います（最大20、最小3）。L2 Tableのメンバーに対して他の席が介入することはありません。
    * **投票:** 各L1席にはGovernance Gameで1票があります。L2席では、そのL2 tableの1票は、そのL2 tableが決めたロジックによって決まります。既定では、そのL2 tableのメンバーの50%超の票です。
    * **Hookの変更:** Hookに対する変更はすべてGovernance Gameを通じて投票されます。Hookの変更が成功するのは、定義された票数がすべての席から支持された場合のみです。通常の投票要件は80%ですが、残高調整率の引き上げには100%が必要です。
    * **報酬:** L1席が信頼できるバリデーターを運用するよう促すため、L1席はコンセンサスに積極的に参加している限り、月次報酬を得ます。
    * **dUNLの基盤:** Genesis Hook Governance Gameに参加するバリデーターは、どのdUNLの基盤にもなるべきです。信頼性の低い席は、他の席によって追い出されるリスクがあるためです。
5. 席の種類が異なることで、セキュリティを損なうことなく、参加アカウントの多様性を高められます。Level 1の席は、コミュニティが信頼できる大きな組織が管理するアカウントに向いており、Layer 2の席はより広い参加者層を許容しますが、その集団としての参加は1票として数えられるため、Level 1の席に置かれた信頼を圧倒することはありません。
6. したがって、このGovernance Gameにより最大400アカウントが参加でき、しかも信頼できるバリデーターを運用してコンセンサスに参加している場合に限り、その参加に対する報酬を受け取れます。

## 6. Xahau Launch Alliance

1. Xahauは、XRPLエコシステムでの構築と支援の実績を持つ5つの独立組織からなる連合によって開発されました。各組織は次のとおりです。
    * XRPL Labs（Xaman）: XUMMウォレット、Hooks、Xahaudの背後にあるソフトウェア開発者であり、Xahau開発における専門知識とコストの大半を担ってきました。
    * GateHub Limited: XRPL発行のステーブルコインを含む、国際的なテクノロジー企業、暗号資産取引所、暗号資産サービス提供者です。
    * Titanium OU:安全なホスティングに特化したITコンサルティングおよびインフラ企業で、XRPL向けインフラの主要提供者のひとつです（Alloy Networksとして）。
    * Evernode Labs Ltd: Evernodeスマートコントラクトプロジェクトの開発者であり、Xahauへの展開を担当しています。
    * Digital Governing OU:会計、監査、法律サービスを提供するグローバル企業に関連する法人です。

## 7. 完全機能かつ分散型でローンチ

1. Xahauは、資金調達を行わずに、完全機能かつ分散型でローンチします。中央集権的な管理はありません。
2. Xahauは、20席あるGovernance Gameのうち8席を埋めた状態でローンチします。
    * 5席はLayer 1席で、Xahau Launch Allianceの各メンバーが1席ずつ埋めます。
    * 3席はLayer 2テーブルで、そのうち1つは取引所の委員会、もう1つはXRPコミュニティプロジェクト、開発者、長期支援者からなる多様な委員会が埋めます。
    * 1つはFYEOであり、Hooksのようなソフトウェアを含むブロックチェーン開発のコード監査・レビューサービスを提供する企業です。

### 資金調達はしない

3. ICOはなく、ローンチに伴う資金調達も行いません。
4. Xahau Allianceの各メンバーは、それぞれが自分の開発活動に独自に資金を投じています。共同の資金調達も、Xahauのさらなる開発のための資金やリソースのプールもありません。ロードマップや「Xahau Foundation」、その他の中央集権的な財務基盤も存在しません。以後の開発は、エコシステム内の独立した参加者の裁量に完全に委ねられます。

### 約束はしない

5. Governance Gameのメンバーは独立した組織です。投票の仕方、リソースの投入、XAHを含む資産の使い方について、相互の合意・取り決め・了解はありません。Validatorとして他のメンバーの目に信頼できる存在であるか、そうでなければGameから投票で外されるかというGameのロジックだけが、バリデーターの行動を決定します。これは、Xahauが利用するXRP Ledger Consensus Protocol（以前: Ripple Protocol Consensus Algorithm）（PoA）機構の本質的な特徴です。

### これ以上の機能追加は不要

6. ソフトウェアは本質的に保守が必要ですが、Xahaudには欠けている基本機能はなく、XRPLのスマートコントラクト用サイドチェーンとして有用であるために、さらなる開発が必須というわけでもありません。
