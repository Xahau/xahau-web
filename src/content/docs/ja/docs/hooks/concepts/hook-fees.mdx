### Hook作成手数料

SetHookトランザクションは作成されたwebassemblyのバイトごとに課金されます。レートはバイトあたり500ドロップです。したがって、1kiBのHookの作成には0.5 XAHかかります。

### Hook実行手数料

HooksがStrongly Executed（[強い実行](/docs/hooks/concepts/weak-and-strong)）される場合、発信トランザクションはStrongly ExecutedのEの費用を発信トランザクションの手数料で支払う必要があります。

Hook実行手数料は、関数`hook`（またはコールバックの場合は`cbak`）の最悪ケースの実行における1 webassembly命令あたり1ドロップで課金されます。したがって、多くのループを持つ小さなHookは高いランタイム手数料を引き起こす可能性があります。

### Fee RPCヘルパー

Hooks Amendmentが有効なレジャー上のトランザクション手数料は、エンドユーザーやウォレットアプリケーションが計算するのに些細ではなくなります。これは強いHooksがトランザクションの発信者によって支払われなければならず、送信アカウントに最大4つの強いHook、受信アカウントに最大4つの強いHookがある可能性があり、さらに関係する他の強いトランザクション利害関係者（一部の特殊なトランザクションタイプの場合に当てはまる）がいる可能性があるためです。さらに、トランザクションがSetHookである場合、パラメーターのサイズ、コードのサイズ、_create_操作か_install_操作かによって手数料のサイズが決まります。

したがって、**すべての**トランザクションを更新されたfee RPCコールを通して実行してから、レジャーに送信することを強く推奨します。

RPCコールを呼び出すには：

1. 作業するHooksノードへのWebSocket接続を開きます。
2. 手数料を知りたいシリアライズされたトランザクションを以下のように作成します：

* `Fee: "0"`
* `SigningPubKey: ""`（つまり：型0x73の0バイトVL。16進数で：`0x7300`）
* トランザクションに**署名しない**でください。

3. 次のようにRPCに16進数ブロブとして送信します：

```json
{"command":"fee", "tx_blob":"<hex blob>"}
```

HTTP POST RPCの場合は次のように送信します：

```json
{"method":"fee", "params": [{"tx_blob":"<hex blob>"}] }
```

レスポンスは次のようになるはずです：

```json
{
  result: {
    drops: {
      base_fee: '130520',
    },
  //...
  },
  type: 'response'
}
```

基本手数料を取得し、トランザクションの`Fee`フィールドとして設定します。これで通常のトランザクション送信プロセスに従って署名して送信します。

`tx_blob`の値が無効または`tx_blob`が見つからない場合、通常のJSONレスポンスが`base_fee`が10として返されます。

### Emission手数料

HooksはFee RPCヘルパーが行うのと同じ計算にアクセスできます。これを使用するには、シリアライズされたトランザクションを含むバッファを引数として[etxn_fee_base](/docs/hooks/functions/emitted-transaction/etxn_fee_base)を呼び出すだけです。RPCコールと同様に、シリアライズされたトランザクションに`Fee`フィールドが存在することを確認する必要があります。値は関係ありません。

`etxn_fee_base`が推奨手数料を返したら、Emit前に[sto_emplace](/docs/hooks/functions/serialization/sto_emplace)を使用してシリアライズされたトランザクションに埋め込むことができます。関連するフィールドは`sfFee`です。
