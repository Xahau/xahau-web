<Aside type="tip" title="Hook設計哲学">
  トランザクションの影響を受けるすべての当事者には、自分のHooksを実行する機会があるべきです。
</Aside>

Transactional Stake Holders（TSH）は、何らかの形でトランザクションに利害を持つ、またはその他の形で影響を受ける当事者です。その利害は _weak_ または _strong_ のいずれかです。トランザクションとの結び付きの度合いによって、その当事者にHooksを実行する権利があるかどうか、そしてその実行費用を誰が支払う必要があるかが決まります。

例:

* 従来の直接XAH **Payment**トランザクションでは、2つのTSHは _発信アカウント_ と _宛先アカウント_ です。
* **SetSignerList**トランザクションでは、TSHは _発信アカウント_ と、署名者リストにアドレスが現れる各アカウント（そのようなアカウントがレジャー上でアクティブな場合）です。
* **OfferCreate**トランザクションでは、発信トランザクションと交差する他のアカウントのOfferはすべてweak TSHであり、weak実行を選択できます。
* **IOUIssuerWeakTSH** Amendmentにより、通貨発行者は、自分の通貨に触れる、または言及する第三者トランザクションで、weak実行にオプトインしている場合にHooksを実行できます。

Xahau上のトランザクションは異種であるため、TSHにはさまざまな形があり、特殊で直感的でないものもあります。時間が経過し、より多くのトランザクションタイプがレジャーに追加されるにつれて、この傾向はさらに強まります。

### WeakとStrong

各TSHは、トランザクションに対してweakまたはstrongのいずれかの接続を持ちます。

**Strong**接続とは:

1. 発信トランザクションがTSH Hookチェーンの実行手数料を支払わなければならない。
2. TSHは実行中に自分のHookから`rollback()`を呼び出して、トランザクション全体をロールバックする権利を持つ。

**Weak**接続とは:

1. 発信トランザクションはTSH Hookチェーンの実行費用を支払い**ません**。
2. TSHは[Collect Call Hooks](/ja/docs/hooks/concepts/collect-call)という機能を通じて、自分自身のHookチェーンの実行費用を支払います。
3. TSHは発信トランザクションの実行前に、アカウントフラグ`asfTshCollect`を設定していなければなりません。
4. TSHは実行中に自分のHookから`rollback()`を呼び出して、トランザクション全体をロールバックする権利を持ち**ません**（ただし、自分のHook状態を変更し、トランザクションを発行することは引き続き可能です）。

### 前か後か

Strong TSHesは、発信トランザクションがレジャーに適用される _前_ にHooksを実行します。これは、トランザクションがまだ適用されていないため、[rollback](/ja/docs/hooks/functions/control/rollback)できることを意味します。これにより、Strong実行されたHooksはトランザクションの発生を完全にブロックできます。

Weak TSHesは、発信トランザクションがレジャーに適用された _後_ にHooksを実行します。これは、[トランザクションメタデータ](/ja/docs/hooks/functions/originating-transaction/meta_slot)にアクセスできるものの、トランザクションの発生を防ぐことはできないことを意味します。

<Aside type="tip">
  Strong実行されたHooksは<a href="ref:hook_again">hook_again</a>を呼び出すことで、発信トランザクションが適用された後にWeak実行として2回目に実行されることができます。
</Aside>

### 実行コンテキスト

`hook(uint32_t)`と`cbak(uint32_t)`の`uint32_t`パラメーターは、Hooks AmendmentからHookへ重要なコンテキスト情報を運びます。

`hook`の実行中:

* 0はHookが _Strongに_ 実行されていることを意味します。
* 1はHookが _Weakに_ 実行されていることを意味します。
* 2は、[hook_again](/ja/docs/hooks/functions/hook-context/hook_again)呼び出しにより、 _Strongに_ 実行された後に _Weakに_ 実行されていることを意味します。

`cbak`の実行中:

* 0は、Hookが発行したトランザクションがレジャーに正常に受け入れられた後に、Hookがコールバックされていることを意味します。
* 1は、Hookが発行したトランザクションがどのレジャーにも適用不能としてマークされた後に、Hookがコールバックされていることを意味します（EmitFailure）。
