---
title: Transaction Results
---
[\[Source\]](https://github.com/ripple/rippled/blob/master/src/ripple/protocol/TER.h)

`rippled` 服务器会使用结果码来总结交易结果，这些结果码会出现在 `engine_result` 和 `meta.TransactionResult` 等字段中。这些代码按不同前缀分成多个类别：

| 类别 | 前缀 | 说明 |
| --- | --- | --- |
| 仅收取成本 | `tec` | 交易没有实现预期目的，但交易费用已被销毁。只有当它出现在已验证账本中时，这个结果才是最终结果。 |
| 失败 | `tef` | 交易不能应用到服务器当前（进行中）的账本，也不能应用到之后任何账本。它可能已经被应用过，或者当前账本状态使其未来不可能再应用。 |
| 本地错误 | `tel` | `rippled` 服务器因本地条件（例如高负载）而出错。如果你提交到其他服务器或稍后重试，可能得到不同响应。 |
| 格式错误交易 | `tem` | 交易无效，原因可能是语法不正确、选项冲突、签名错误或其他问题。 |
| 可重试 | `ter` | 交易当前无法应用，但未来某个账本中可能成功。 |
| 成功 | `tes` | （不是错误）交易成功。只有出现在已验证账本中时，这个结果才是最终结果。 |

`rippled` 服务器会自动重试失败的交易。不要仅凭一个暂时失败结果就断定交易彻底失败了。除非成功或失败已经成为最终状态，否则交易之后仍可能成功。

**警告：** 交易的 provisional result code 可能与最终结果不同。暂时成功的交易最终可能失败，暂时失败的交易也可能最终成功。暂时失败的交易最终也可能以不同代码失败。关于如何判断结果是否最终，请参见 result finality。

本地错误（`tel`）与格式错误交易（`tem`）之间的区别，取决于协议层规则。例如，协议本身并没有规定交易中最多能包含多少条 paths，但服务器可以自行设定可处理的最大 paths 数量。如果两个服务器配置不同，其中一个可能对包含大量 paths 的交易返回 `tel`，而另一个服务器则可能成功处理它。如果有足够多的服务器能处理这笔交易，使其通过共识，那么它仍然可能被纳入已验证账本。

相比之下，`tem` 错误意味着无论任何服务器、任何设置，都无法应用该交易。要么交易违反了协议规则，要么它存在不可接受的歧义，要么完全没有意义。格式错误交易只有在协议规则发生变化时才可能变得有效；例如，当某个新特性被采纳后，使用该特性的交易在旧版本服务器看来可能仍是 malformed。

### Immediate Response

[submit method][] 的响应中会包含一个来自本地 `rippled` 服务器的 provisional result，表示交易在本地处理时发生了什么。

`submit` 响应包含以下字段：

| 字段 | 值 | 说明 |
| --- | --- | --- |
| `engine_result` | String | 表示交易结果的代码，例如 `tecPATH_DRY`。 |
| `engine_result_code` | Signed Integer | 对应 `engine_result` 的数值。其具体值可能在未通知的情况下改变。 |
| `engine_result_message` | String | 人类可读的信息，用来帮助开发者诊断问题；也可能在未通知的情况下改变。 |

如果本地提交与应用过程中没有出错，响应通常如下：

```js
"engine_result": "tesSUCCESS",
"engine_result_code": 0,
"engine_result_message": "The transaction was applied. Only final in a validated ledger."
```

**注意：** 在这个阶段看到成功结果，并不表示交易已经完全成功；它只表示交易被成功应用到了本地服务器维护的账本临时版本中。此时的失败结果同样也只是暂时结果，后续仍可能变化。
