Cross-Chain Export lets Xahau accounts create transactions for other XRPL-compatible chains (for example, XRPL mainnet) that the active Xahau validator set collectively multisigns.

_(Requires the [Export amendment](/docs/features/amendments/#export).)_

## User Guide

### Overview

The Export feature coordinates a target-chain transaction through the Xahau validator set:

1. Submit an Export transaction on Xahau containing the inner transaction you want executed on the target chain
2. Validators collectively sign the inner transaction during the consensus round
3. Extract the multisigned blob from the transaction metadata
4. Submit it to the target chain
5. Import the proof of execution back to Xahau to complete the round-trip

<Aside type="note">
Validator quorum is normally expected in the **same ledger** the Export is submitted. If quorum is not reached, the Export retries until its top-level `LastLedgerSequence`.
</Aside>

### Prerequisites

- The target chain must support XRPL-style transactions, tickets, signer lists, and multisigning.
- The target-chain account must be the same AccountID/classic address as the outer Xahau `Account`; the inner `ExportedTxn.Account` must match it.
- The target-chain account must have a SignerList whose signer accounts are derived from the active Xahau validator public keys, using the active validator list for the exporting network.
- The target SignerQuorum and signer weights must accept the validator multisig set that Xahau will produce.
- Keep the target SignerList aligned with validator-set changes. A stale target SignerList can make future exported blobs fail on the target chain.
- For XRPL-mainnet-style target transactions with no `NetworkID`, the exporting Xahau network must have a nonzero local `NETWORK_ID`.

### Quick Start

#### 1. Set Up the Target Chain Account

On the target chain (e.g. XRPL), the account needs a [SignerList](https://xrpl.org/docs/references/protocol/ledger-data/ledger-entry-types/signerlist) containing the active Xahau validator keys as signers.

#### 2. Create Tickets on the Target Chain

Exported transactions use [TicketCreate transactions](/docs/protocol-reference/transactions/transaction-types/ticketcreate) and `TicketSequence` (not regular sequence numbers) to avoid sequence jams:

```json
{
  "TransactionType": "TicketCreate",
  "Account": "rYourAccount...",
  "TicketCount": 1
}
```

#### 3. Submit the Export on Xahau

```json
{
  "TransactionType": "Export",
  "Account": "rYourAccount...",
  "Fee": "1000000",
  "LastLedgerSequence": 12345680,
  "ExportedTxn": {
    "TransactionType": "Payment",
    "Account": "rYourAccount...",
    "Destination": "rRecipient...",
    "Amount": "1000000",
    "Fee": "1000",
    "Sequence": 0,
    "TicketSequence": 2,
    "Flags": 2147483648,
    "SigningPubKey": "",
    "LastLedgerSequence": 98765500
  }
}
```

The top-level `LastLedgerSequence` controls how long Xahau retries the Export while waiting for validator signatures. The inner `ExportedTxn.LastLedgerSequence` is part of the target-chain transaction and does not expire the Xahau Export. Both values are absolute ledger indexes; calculate them from the current Xahau and target-chain ledger indexes before submitting.

#### 4. Extract the Signed Blob

On success, the transaction metadata contains an `ExportResult` with the complete multisigned inner transaction including the `Signers` array. Serialize `ExportResult.ExportedTxn` and submit it to the target chain.

Before relying on production exports, test that the `ExportResult.ExportedTxn.Signers[].Signer.Account` values are authorized by the target-chain SignerList.

#### 5. Import Back (Round-Trip)

After execution on the target chain, build an XPOP from the target chain's validated ledger data and submit it as an [Import transaction](/docs/protocol-reference/transactions/transaction-types/import) on Xahau to consume the shadow ticket. If the signed blob was not and will not be submitted to the target chain, cancel the shadow ticket instead.

### Shadow Tickets

Every successful Export that carries an `ExportedTxn` creates a **shadow ticket** — an on-ledger record keyed by `(Account, TicketSequence)` that tracks outstanding exports and provides replay protection.

- **Created** on `tesSUCCESS` of the Export
- **Consumed** when the XPOP is imported back
- **Cancelled** via Export with `CancelTicketSequence` or `xport_cancel()` from hooks

### Retriable Semantics

If validator quorum isn't reached in the submission ledger, the Export automatically retries each ledger until `LastLedgerSequence`. If quorum is never reached, the Export expires with `tecEXPORT_EXPIRED` — the sequence is consumed cleanly and the account is not blocked.

The open ledger can contain up to 8 pending exports total across all accounts. Separately, each account can hold up to 8 outstanding shadow tickets. If either limit is reached, new Export transactions fail with `tecDIR_FULL`.

### Hook API (xport)

Hooks can programmatically export transactions using the `xport` API:

```c
if (xport_reserve(1) < 0)
    rollback(SBUF("Export reserve failed"), 1);

// ... build inner tx blob ...

uint8_t hash[32];
int64_t result = xport(SBUF(hash), SBUF(tx_blob));  // returns inner tx hash
if (result < 0)
    rollback(SBUF("Export failed"), 1);

xport_cancel(ticket_seq);  // cancel a shadow ticket
```
